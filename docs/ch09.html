Ch9. Consistency and Consensus

단 하나의 복사본이라는 환상(linearizability)에서 출발해, causality와 Lamport timestamp를 지나 total order broadcast = consensus에 도달하는 여정 — 그리고 2PC의 함정과 ZooKeeper라는 "consensus 아웃소싱"

챕터 개요

Chapter 7의 트랜잭션이 여러 fault를 "없는 것처럼" 만들어주는 범용 추상화였다면, 이 챕터는 분산 시스템을 위한 범용 일관성 추상화를 찾는다. 8장에서 봤듯 패킷은 유실·지연되고 시계는 어긋나며 노드는 언제든 멈춘다 — 그럼에도 애플리케이션이 이를 일일이 신경 쓰지 않게 해줄 강력한 보장을 만들 수 있을까?

가장 널리 쓰이는 eventual consistency는 매우 약하다 — "언젠가 수렴한다"고만 말할 뿐 언제인지는 말하지 않고, 그 전까지 읽기는 무엇이든 반환할 수 있다(자신이 방금 쓴 값조차 못 읽을 수 있다). 이런 미묘함은 네트워크 장애나 높은 동시성에서만 드러나 버그를 찾기 어렵다. 그래서 이 챕터는 더 강한 보장을 단계적으로 쌓는다: linearizability → ordering & causality → total order broadcast → atomic commit(2PC)과 fault-tolerant consensus.

결론부의 핵심 통찰 — linearizable cas register, total order broadcast, atomic commit, lock/lease, uniqueness constraint… 이 모든 문제는 사실상 consensus와 동등(equivalent)하다. 하나를 풀면 나머지도 풀 수 있다.

핵심 개념 정리

Linearizability — 단 하나의 복사본이라는 환상

정의: 시스템이 마치 데이터의 복사본이 단 하나만 있고, 모든 연산이 그 복사본에 원자적으로 수행되는 것처럼 보이게 하는 것 (atomic/strong/immediate/external consistency). 핵심은 recency guarantee — 어떤 읽기가 최신 값을 반환한 순간부터, 이후의 모든 읽기(다른 클라이언트 포함)도 그 값(또는 더 나중의 값)을 반환해야 한다. 검증 방법: 각 연산이 요청~응답 사이의 어느 한 시점에 원자적으로 발효된 것처럼 모든 연산을 하나의 timeline에 배치할 수 있으면 linearizable하다. 동시 쓰기 경합까지 다루려면 compare-and-set(cas) — "현재 값이 v_old면 v_new로, 아니면 실패" — 도 register 의미론에 포함시킨다.

심층 설명 — linearizability vs serializability: 이름이 비슷해 자주 혼동되지만 전혀 다른 보장이다. serializability는 트랜잭션(여러 객체를 읽고 쓰는 묶음)의 isolation 속성 — 트랜잭션들이 어떤 순차 순서로 실행된 것과 같은 결과면 되고, 그 순서가 실제 실행 시간 순서와 달라도 된다(과거 snapshot에서 읽어도 OK). linearizability는 단일 객체 읽기·쓰기의 최신성(recency) 보장 — 실시간 순서를 존중해야 하지만 트랜잭션 묶음 개념은 없다(write skew를 막지 못한다). 둘을 모두 보장하면 strict serializability다. 구현별로: 2PL과 actual serial execution은 보통 linearizable하지만, SSI(serializable snapshot isolation)는 linearizable하지 않다 — 일관된 snapshot에서 읽는 것이 요점이라 snapshot 이후의 쓰기를 보지 못하기 때문이다.

linearizability에 의존하는 곳:

  • Locking과 leader election: "leader는 하나"를 보장하는 분산 lock은 모든 노드가 누가 lock을 쥐었는지 동의해야 하므로 linearizable해야 한다 — ZooKeeper 등 coordination 서비스의 용도.
  • Uniqueness constraints: username·이메일 유일성, 잔고 ≥ 0, 재고 초과 판매 방지 — "동시에 같은 값을 쓰려는 둘 중 하나만 성공"은 lock/cas와 같은 구조다.
  • Cross-channel timing dependencies: 통신 채널이 둘 이상이면 미묘한 경쟁이 생긴다. 이미지 리사이즈 예 — 웹 서버가 ① 원본을 storage에 쓰고 ② message queue로 리사이저에게 지시한다. queue(채널 2)가 storage 복제(채널 1)보다 빠르면 리사이저는 옛 이미지나 빈 상태를 읽어 영구 불일치를 만든다. storage가 linearizable하면 문제없다.
복제 방식 Linearizable? 이유
Single-leader 잠재적으로 가능 leader나 동기 복제 follower에서 읽으면 — 단 snapshot isolation 사용, split brain(자칭 leader) 시 깨진다
Consensus 알고리즘 가능 split brain·stale replica 방지 내장 (ZooKeeper, etcd)
Multi-leader 불가 여러 노드가 동시에 쓰기를 받고 비동기 전파·충돌 — "단일 복사본" 환상과 모순
Leaderless (Dynamo류) 아마도 불가 LWW + time-of-day clock, sloppy quorum은 물론 — strict quorum(w+r>n)조차 쓰기가 replica에 퍼지는 도중엔 recency가 깨질 수 있다

보완책: reader가 반환 전 read repair를 동기 수행하고 writer가 쓰기 전 quorum 최신 상태를 읽으면 linearizable read/write는 가능하다(Attiya, Bar-Noy, Dolev 알고리즘) — 하지만 cas는 불가(consensus 필요)하고 성능 손해가 커서 Cassandra 등은 하지 않는다. 결론: Dynamo류는 linearizability를 제공하지 않는다고 가정하는 것이 안전하다.

대가 — CAP theorem: 네트워크가 단절(partition)되면 양자택일이다. linearizability를 지키려면 단절된 쪽 replica는 요청을 거부/대기해야 하고(unavailable, CP), 각 쪽이 독립적으로 응답하면 available하지만 non-linearizable하다(AP — 다중 데이터센터 multi-leader가 이 예). 올바른 독법은 "either Consistent or Available when Partitioned" — partition은 선택지가 아니라 일어나는 fault다. "3개 중 2개"라는 통속적 표현은 오해를 부르고, CAP은 하나의 일관성 모델(linearizability)과 하나의 fault(partition)만 다룰 뿐 네트워크 지연·죽은 노드 등은 아무것도 말해주지 않는다 — 역사적 영향(NoSQL 운동)은 컸지만 실용적 가치는 제한적이다.

더 근본적 이유는 성능이다 — 현대 멀티코어 CPU의 RAM조차 linearizable하지 않다. 코어마다 store buffer·cache가 있어 fence 명령 없이는 다른 코어의 쓰기가 바로 안 보인다. 코어 간 네트워크가 끊길 일은 없으니 이유는 fault tolerance가 아니라 순전히 성능이다. 분산 DB도 마찬가지 — linearizability는 partition 때만이 아니라 항상 느리다 (Attiya·Welch: 응답 시간을 네트워크 지연 비례 하한 아래로 낮추려면 linearizability를 포기해야 한다).

Ordering Guarantees — 순서와 인과성

  • 순서가 반복 등장하는 이유는 causality 보존 때문이다 — 질문이 답변보다 먼저, row 생성 후 갱신, prefix 읽기·write skew 모두 인과 위반 문제였다. 인과로 연결된 사건들의 순서가 causal order(happens-before)다.
  • Total vs partial order: linearizability는 모든 연산의 total order(단일 복사본 timeline)를 만들고, causality는 partial order만 정의한다 — concurrent한 연산끼리는 비교 불가(갈라졌다 병합되는 git 히스토리처럼).
  • Linearizability ⇒ causality — 역은 성립하지 않는다. causal consistency는 linearizability 없이 달성 가능하며, 네트워크 지연에 둔감하고 partition에서도 available한 일관성 모델 중 가장 강하다. 많은 경우 linearizability가 필요해 보여도 실제로 필요한 것은 causality다(연구 활발, 프로덕션 일반화 전). 이를 위해 시스템은 happens-before를 추적해야 한다 — version vector가 그 도구다.
  • Lamport timestamp: 각 노드가 (counter, node ID)를 발급하고, 모든 노드·클라이언트가 지금까지 본 최대 counter를 추적·전파하며 더 큰 값을 보면 자기 counter를 끌어올린다. 인과 의존은 반드시 timestamp 증가를 동반하므로 causality와 consistent한 total order가 생긴다. version vector와의 차이: version vector는 concurrent/causal을 구별하고, Lamport timestamp는 total order를 강제하는 대신 구별을 못 한다(더 가볍다) — 용도가 다르다.
심층 설명 — Lamport timestamp가 uniqueness 보장엔 부족한 이유 → total order broadcast: 두 사용자가 동시에 같은 username을 만들려 할 때 "timestamp 낮은 쪽 승리"는 사후에는 완벽히 판정된다. 문제는 지금 요청을 처리해야 하는 노드다 — 다른 어떤 노드가 더 낮은 timestamp로 같은 username을 claim하고 있는지 알 방법이 없고, 확실히 하려면 모든 노드에 물어봐야 하며 한 노드라도 응답이 없으면 시스템이 멈춘다(fault tolerance 상실). 본질: 연산의 total order가 존재하는 것만으로는 부족하고, 그 순서가 언제 확정(finalized)되는지 — "내 연산 앞에 더는 아무것도 끼어들 수 없다" — 를 알아야 한다. 이 확정성까지 주는 것이 total order broadcast다: ① reliable delivery(하나에 전달되면 전부에), ② totally ordered delivery(전 노드 동일 순서), 그리고 순서는 전달 시점에 고정되어 소급 삽입이 불가능하다. 이 위에서 linearizable cas를 만들 수 있다 — username claim 메시지를 log에 append하고, 자기 메시지가 돌아올 때까지 log를 읽어 내 claim이 그 username의 첫 claim인지 확인하면 된다(모든 노드가 같은 순서로 같은 결론). 역방향으로 linearizable increment-and-get register로 빈틈 없는 연속 번호를 발급하면 total order broadcast를 만들 수 있다 — Lamport timestamp와 달리 번호에 구멍이 없어 "앞 번호를 다 받을 때까지 기다리기"가 가능하다. 결국 둘 다 consensus와 동등한 문제다.
  • Total order broadcast의 용도: DB 복제 — 같은 쓰기를 같은 순서로 처리하면 replica들이 일관된다 (state machine replication; log 기반 복제의 원리); serializable 트랜잭션(stored procedure를 같은 순서로 실행); log(replication log·transaction log·WAL — 메시지 추가 = append); lock 서비스의 fencing token(lock 요청을 log 순서대로 번호 매김 — ZooKeeper의 zxid).
  • Linearizable read의 세 방법: ① 읽기 요청 자체를 log에 append해 자기 차례에 읽기, ② 최신 log 위치까지의 메시지를 기다린 뒤 읽기(ZooKeeper의 sync()), ③ 동기 복제된 replica에서 읽기.

Atomic Commit과 2PC — 분산 트랜잭션

단일 노드의 commit은 디스크에 commit record가 쓰이는 순간이라는 단일 장치의 단일 사건이다. 여러 노드가 얽히면 일부만 commit되는 반쪽 상태를 막아야 한다 — commit은 되돌릴 수 없기 때문이다(read committed의 근간). 2PC(two-phase commit)는 이를 위해 coordinator(transaction manager)를 도입한다:

단계 동작 point of no return
Phase 1 — Prepare coordinator가 전 participant에게 "commit 가능?" 질의 participant의 "yes"는 약속 — 데이터를 디스크에 쓰고 검사를 마쳐, crash 후에도 반드시 commit할 수 있음을 보증하고 abort 권리를 포기
Phase 2 — Commit/Abort 전원 yes면 commit, 하나라도 no면 abort 전파 coordinator가 결정을 자기 디스크 log에 쓰는 순간이 commit point — 이후엔 응답할 때까지 영원히 재시도
심층 설명 — 2PC의 coordinator 장애가 치명적인 이유 vs 진짜 consensus: participant가 "yes"를 보낸 후 coordinator가 죽으면 그 participant는 in doubt(uncertain) 상태가 된다 — 혼자 abort할 수 없고(약속을 했으므로), 혼자 commit할 수도 없다(다른 참여자에게 이미 abort가 전달됐을지 모르므로). 유일한 진실은 coordinator의 디스크 log이고, participant는 lock을 쥔 채 coordinator 복구를 무한정 기다려야 한다 — 그래서 2PC는 "anti-availability protocol"이라 불린다. nonblocking을 노린 3PC도 있지만 unbounded delay 환경에서는 perfect failure detector가 필요해 성립하지 않는다. consensus의 언어로 보면: 2PC는 uniform agreement·integrity·validity는 만족하지만 termination(crash 안 한 노드는 결국 결정한다)을 위반한다. 진짜 fault-tolerant consensus(Raft 등)는 다르다 — coordinator에 해당하는 leader가 죽으면 새 epoch로 새 leader를 선출하고, 전원 만장일치 대신 과반 quorum만 요구하며, 선출 quorum과 제안 quorum이 반드시 겹치기 때문에 낡은 leader의 결정은 자동으로 거부된다. 요컨대 2PC는 "leader가 고정이고 죽으면 전원이 기다리는" 반쪽 합의, consensus는 "leader를 갈아치우며 과반으로 전진하는" 완전한 합의다.

실무의 분산 트랜잭션 — XA: heterogeneous 시스템(서로 다른 DB, message broker 등) 간 2PC의 표준(1991~). coordinator는 보통 애플리케이션 프로세스 안의 라이브러리로, 자기 로컬 디스크에 결정을 log한다. 문제점:

  • Locks held while in doubt: in-doubt 트랜잭션의 row lock은 해소까지 유지 — coordinator가 20분 죽으면 20분, log를 잃으면 영원히. 그동안 다른 트랜잭션이 막혀 앱 상당 부분이 마비된다.
  • Coordinator SPOF: 많은 구현이 기본 무복제 단일 머신 — 전체의 단일 장애점. 또 log가 앱 서버 디스크에 있어 "stateless" 서버 모델이 깨진다.
  • Orphaned in-doubt transactions: log 유실·손상 시 관리자가 수동으로 commit/rollback해야 하며, 최후 수단인 heuristic decision(participant의 일방 결정)은 원자성을 깰 수 있다.
  • XA는 최소공통분모라 deadlock 감지·SSI와 함께 동작하기 어렵고, 어느 한 부분의 실패가 전체를 실패시키는 장애 증폭기가 된다. (MySQL 분산 트랜잭션이 단일 노드 대비 10배 느리다는 보고도.)

Fault-Tolerant Consensus — 진짜 합의

형식화: 노드들이 값을 propose하고 알고리즘이 하나를 decide한다. 만족할 속성 — uniform agreement(두 노드가 다르게 결정하지 않음), integrity(두 번 결정하지 않음), validity(결정값은 누군가 제안한 값), termination(crash 안 한 노드는 결국 결정 — fault tolerance의 핵심 liveness; 과반 생존 필요, Byzantine fault는 보통 가정하지 않음).

FLP result — asynchronous model(시계·timeout 없음)에서는 노드 crash 가능성이 있으면 항상 종료가 보장되는 deterministic consensus 알고리즘이 존재하지 않는다는 증명. 하지만 timeout이나 randomness를 쓸 수 있는 현실 시스템에서 consensus는 풀린다 — FLP는 이론적 한계이지 실무적 불가능이 아니다.
  • 알고리즘: Paxos(Multi-Paxos), Raft, Zab(ZooKeeper), VSR — 대부분 단일 값이 아니라 값의 시퀀스를 결정한다, 즉 total order broadcast를 구현한다(각 라운드가 "다음 메시지"의 합의).
  • Epoch numbering + quorum: "leader 영원히 유일" 대신 epoch 번호(Paxos ballot, Raft term, VSR view) 안에서의 유일성만 약속한다. leader가 죽은 것 같으면 epoch를 올려 새로 선출. 결정마다 두 라운드의 투표 — ① leader 선출, ② 제안 승인 — 가 있고 두 quorum(과반)은 반드시 겹치므로, 더 높은 epoch가 존재하면 제안 투표에서 드러나 낡은 leader의 제안이 거부된다. 2PC와의 차이: coordinator는 선출되지 않고 전원 yes가 필요하지만, consensus는 과반이면 되고 leader 교체 후 상태 회복 절차가 있다.
  • 한계·비용: ① 제안 투표는 사실상 동기 복제(성능 비용 — DB들이 비동기 복제를 택하는 이유); ② 과반(3+ 또는 5+ 노드) 필수 — 과반이 죽으면 정지; ③ dynamic membership은 훨씬 복잡; ④ timeout 기반 leader 감지라 지연 변동이 큰 환경(지리적 분산)에선 잦은 헛 선출; ⑤ 특정 네트워크 패턴에 민감 — Raft는 한 링크만 불안정해도 leadership이 두 노드 사이를 오가는 edge case가 알려져 있다.

ZooKeeper / etcd — "consensus의 아웃소싱"

분산 key-value 저장소지만 직접 쓰기보다 HBase, Hadoop YARN, Kafka 등이 간접적으로 의존하는 coordination 서비스다. 메모리에 들어가는 소량 데이터를 consensus(total order broadcast)로 전 노드에 복제하며, 다음을 조합해 제공한다:

기능 내용
Linearizable atomic operations atomic cas로 lock/lease 구현
Total ordering of operations 모든 연산에 단조 증가 zxid 부여 — fencing token으로 사용 가능
Failure detection 세션 heartbeat — 세션이 끊기면 ephemeral node가 자동 삭제(lock 자동 해제)
Change notifications watch — polling 없이 다른 클라이언트의 변화를 통지

사용 패턴: leader election, partition assignment(rebalancing 포함) — 애플리케이션이 직접 consensus를 구현하지 않고 아웃소싱한다. 보통 3~5개 고정 노드에서 수천 클라이언트를 지원하며, 천천히 변하는 데이터(수분~수시간) 전용이지 런타임 상태 저장용이 아니다. 파생 용도: service discovery, membership service(누가 살아 있는 멤버인지 — 판정이 틀릴 수는 있어도 전원이 동의한다는 것이 가치). 마무리 통찰: leader가 죽었을 때의 선택지는 ① 사람 개입 대기, ② 영원히 대기, ③ 알고리즘적 consensus뿐이다 — 그리고 causal consistency로 충분하고 충돌을 병합할 수 있다면 consensus 없이도 잘 산다.

쉬운 예시 / 비유

Linearizability — 원본이 하나뿐인 장부

사본이 몇 개든, 모두가 마치 금고 속 원본 장부 하나에만 쓰고 읽는 것처럼 보여야 한다. 누군가 최신 기입을 봤다면 그 뒤에 보는 사람은 절대 그 이전 페이지 상태를 보면 안 된다 — 봤다면 사본이 여럿임이 들통난 것.

Recency 위반 — 축구 결승 알림

Alice가 폰에서 결승 결과를 보고 환호했는데, 옆의 Bob이 새로고침하니 "경기 진행 중"이라고 나온다. Alice가 말해준 뒤에 조회했는데도 과거를 본다 — stale replica에 붙은 것이며, linearizability가 금지하는 상황이다.

Serializability vs linearizability — 소설 vs 생중계

serializability는 "여러 사건을 어떤 일관된 이야기 순서로 재구성할 수 있는가"(순서가 실제와 달라도 됨), linearizability는 "지금 이 순간의 최신 상태를 보는가"(생중계). 소설이면서 생중계이면 strict serializability다.

Cross-channel — 전화와 택배

서류를 택배(채널 1)로 보내고 전화(채널 2)로 "지금 도착했지? 처리해줘"라고 말한다. 전화가 택배보다 빠르면 상대는 없는 서류를 처리하려 든다 — 이미지 리사이저가 옛 이미지를 읽는 것과 같은 구조.

CAP — 전용선이 끊긴 두 은행 지점

두 지점 간 전용선이 끊겼다. 둘 다 계속 영업하면(available) 잔고가 어긋날 수 있고(non-linearizable), 장부 일치를 고집하면(consistent) 한쪽은 창구를 닫아야 한다(unavailable). "단절되었을 때"의 양자택일이지, 평소 3개 중 2개를 고르는 게 아니다.

Causal order — 답장이 질문보다 먼저 보이는 채팅방

질문("서버 내렸어?")보다 답("아니, 방금 복구했어")이 먼저 보이면 대화가 미친 것처럼 보인다. 전역 순서까지는 필요 없어도 원인이 결과보다 먼저는 지켜야 한다 — 이것이 causal consistency이고, 단일 timeline을 요구하는 linearizability보다 약하다.

Lamport timestamp — 회의록의 최대 번호 이어받기

부서마다 자체 일련번호로 문서를 만들되, 남의 문서를 볼 때마다 자기 카운터를 그 번호 이상으로 끌어올린다. 참고한 문서는 반드시 더 큰 번호가 된다 — 인과와 일치하는 전역 순서. 하지만 "내 번호보다 작은 문서가 더는 안 나온다"는 보장은 없다 — 어느 부서가 아직 발송 안 한 낮은 번호 문서를 쥐고 있을지 모른다.

Total order broadcast — 창구의 단일 대기줄

모든 요청이 하나의 줄에 서고, 전 창구가 같은 줄 순서로 처리한다. 내 차례가 "확정"되는 순간(앞에 더는 아무도 끼어들 수 없음)이 오기에, "이 username은 내가 처음 claim했나"를 누구나 같은 답으로 판정할 수 있다.

2PC — 결혼식 주례

주례(coordinator)가 신랑·신부(participants)에게 "서약하시겠습니까?" 묻고(prepare), "네"라고 답하면 번복 불가다. 양쪽 다 "네"면 성혼 선언(commit). 문제: "네" 직후 주례가 기절하면 둘은 결혼인지 아닌지 모른 채 주례가 깨어날 때까지 제단에 묶여 있어야 한다(in doubt) — 스스로 결정할 수 없다.

Consensus — 회차가 붙은 반장 선거

"반장은 영원히 하나" 대신 "제N대 반장은 하나"만 약속한다. 반장의 결정마다 과반의 동의 도장이 필요한데, 그 과반 중 누군가는 더 최신 선거(제N+1대)를 알고 있을 수밖에 없다(과반끼리는 반드시 겹치므로) — 낡은 반장의 결재는 자동 거부된다.

ZooKeeper — 사무실의 게시판 + 출입기록부

팀들이 직접 투표 시스템을 만드는 대신 관리실(ZooKeeper)에 lock 신청·리더 등록·좌석 배정을 맡긴다. 퇴실하면(세션 종료) 그 사람 명의의 임시 게시물(ephemeral node)은 자동으로 떼어지고, 게시판이 바뀌면 구독자에게 알림(watch)이 간다 — consensus의 아웃소싱.

한 줄 요약

분산 시스템의 강한 보장은 linearizability(단일 복사본처럼 보이는 recency 보장, 단 느리고 partition 시 가용성 포기)에서 출발해, 더 싸게 얻을 수 있는 causal consistency와 Lamport timestamp를 지나 — "순서가 언제 확정되는가"라는 질문 끝에 total order broadcast = consensus에 도달하며, 2PC는 coordinator가 죽으면 멈추는 반쪽 해법이고, Raft/Paxos류의 진짜 consensus(epoch + 겹치는 quorum)를 ZooKeeper 같은 서비스로 아웃소싱하는 것이 실무의 정답이다.
