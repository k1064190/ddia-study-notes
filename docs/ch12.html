Ch12. The Future of Data Systems

만능 도구는 없다 — system of record와 derived data를 구분하고, 로그 + idempotence로 파생 데이터를 엮는 unbundled database(dataflow 중심 설계), 그리고 정확성은 end-to-end로, 일관성은 timeliness와 integrity로, 마지막으로 데이터를 다루는 자의 윤리

챕터 개요

앞선 장들이 데이터 시스템이 현재 어떻게 동작하는가를 분석했다면, 이 마지막 장은 저자(Martin Kleppmann)의 견해를 담아 어떻게 되어야 하는가를 탐색한다. 목표는 올바르고(correct), 확장 가능하며(scalable), 유지보수 가능한(maintainable) 시스템을 짓는 것이며, 이를 위해 복제·파티셔닝· 트랜잭션·배치·스트림 처리를 하나의 큰 그림으로 엮는다.

출발점은 냉정한 현실이다 — 모든 요구를 만족하는 단일 도구는 없다(no one-size-fits-all). 실무는 여러 저장소·인덱스·캐시·분석 시스템을 조합해 쓸 수밖에 없고, 남는 문제는 이 이질적 조각들을 어떻게 하나의 일관된 시스템으로 통합하는가 — data integration이다. 이 장은 data integration → unbundling databases → aiming for correctness → trust but verify → doing the right thing(윤리)로 전개된다.

결론부의 핵심 통찰 — 미래의 데이터 시스템은 하나의 거대한 DB가 아니라, 로그로 연결된 파생 데이터의 흐름(dataflow)으로 구성된 unbundled database이며, 정확성은 어느 한 계층이 아니라 end-to-end로만 완성된다.

핵심 개념 정리

Data Integration — 파생 데이터를 동기화하기

어떤 저장소·인덱스·처리 시스템도 모든 접근 패턴을 효율적으로 감당하지 못한다. OLTP DB, full-text search index, column 분석 DB, 캐시, 그래프 인덱스, ML 모델을 동시에 쓰게 되고, 그러면 같은 데이터가 여러 곳에 저장되어 일관되게 유지해야 하는 문제가 생긴다. 이때 핵심 관점이 데이터를 두 종류로 나누는 것이다 — system of record(source of truth, 새 데이터가 처음 기록되는 권위 있는 원본)와 derived data(index, cache, materialized view, ML model 등 원본을 변환해 만든, 잃어도 재생성 가능한 중복).

여러 파생 뷰를 원본과 동기화하는 도구가 앞 두 장의 batch·stream 처리다. 원본에 쓰기가 일어나면 그 변화를 downstream 인덱스·캐시·모델로 전파해 파생 데이터를 최신으로 유지한다. 이때 이벤트의 total order(로그)가 충돌을 막는 열쇠다 — 서로 다른 파생 시스템이 쓰기를 다른 순서로 적용하면 영구히 불일치하지만, 모든 쓰기를 하나의 로그로 묶어 같은 순서로 소비하면 결정론적으로 같은 상태에 수렴한다(state machine replication).

단, single total order를 매기는 것은 처리량이 단일 머신을 넘거나 지리적 분산 데이터센터가 얽히면 어렵거나 비싸다 — 결국 consensus/leader 문제이며 (9장), causality만 보존하고 total order를 포기해야 할 수도 있다. 서로 다른 스트림 간 순서, offline·이동 중 클라이언트 이벤트의 순서는 근본적으로 미해결 영역이다.

Derived state vs distributed transactions: 여러 시스템을 일관되게 유지하는 전통적 방법은 distributed transaction(2PC)이지만 성능·운영상 취약하다. 대안이 log 기반 파생 + idempotence다.

Distributed transaction (2PC) Log-based derivation
결합 tight coupling (atomic commit, 동기) loose coupling (asynchronous)
순서 조정 mutual exclusion(lock) 로그의 total order
실패 대응 atomic 2PC — coordinator SPOF idempotence + 재시도로 복구
최신성 linearizable(즉시 일관) eventually consistent(지연 존재)

Unbundling Databases — 데이터베이스를 분해하기

전통적 DB는 secondary index, materialized view, replication log, full-text search를 하나의 통합 제품 안에 감춰 제공한다. Unix가 작은 도구를 파이프로 조립하듯, 이 기능들을 로그를 축으로 분해(unbundling)하면 각자를 특화 도구로 최적화하고 원하는 대로 조립할 수 있다. 통합에는 두 학파가 있다 — federated database(unified read)는 여러 이질 저장소 위에 통합된 읽기 인터페이스(polystore)를 제공하고, unbundled database(unified write)는 로그로 쓰기를 통합하고 거기서 여러 저장소를 파생시킨다 — 이 장의 초점은 후자다.

심층 설명 — unbundling database와 dataflow 중심 설계: 파생 데이터를 만드는 규칙(secondary index를 만드는 함수, cache를 갱신하는 함수, ML 모델을 학습하는 함수)은 사실 애플리케이션 특화된 파생 함수(derivation function)다. 이 세계에서 stream processor는 그 함수들을 실행하며 시스템들을 연결하는 접착제(glue)이자, unbundled DB의 trigger·stored procedure·materialized view 유지 엔진에 해당한다. 설계의 비유는 spreadsheet다 — 한 셀을 바꾸면 그것을 참조하는 모든 셀이 자동으로 재계산되듯, 원본이 바뀌면 의존하는 모든 파생 뷰가 저절로 갱신되게 짠다(reactive). 이를 위해 쓰기는 불변 이벤트로 로그에 append하고(events), 읽기는 그 이벤트들로부터 파생된 materialized view나 on-the-fly 계산으로 답한다(views). write path(쓰기 시 미리 계산)와 read path(요청 시 계산) 사이에서 어디까지 미리 계산할지가 성능 트레이드오프의 본질이며, 클라이언트는 read를 고정된 쿼리가 아니라 변화를 구독하는(subscribe to changes) 살아있는 materialized view로 다룬다.
심층 설명 — end-to-end argument와 idempotence 기반 exactly-once: 실무의 "exactly-once"는 물리적으로 정확히 한 번 실행이 아니라 effectively-once — 결과가 한 번 처리한 것과 같아지게 만드는 것이고, 수단은 idempotence + 중복 제거(dedup)다. 그런데 중복 제거가 제대로 되려면 operation을 유일 식별하는 ID가 필요하고, 그 ID는 최종 사용자 요청에서부터 끝단까지 end-to-end로 전달돼야 한다. 결제 요청에 클라이언트가 만든 request ID를 실으면, 네트워크 재시도로 요청이 두 번 도착해도 서버는 같은 ID를 보고 한 번만 처리한다. end-to-end argument(Saltzer, Reed, Clark)의 요지: 중복 없는 정확한 처리 같은 보장은 통신 경로의 양 끝점에서만 완전히 구현되며, TCP 재전송이나 DB 트랜잭션 같은 하위 계층의 부분 보장은 도움은 되나 충분하지 않다(앱 재시작·중복 제출까지 막지 못하므로). transaction은 사라지지 않고 다르게 조립될 뿐이다.

Aiming for Correctness — 정확성을 겨냥하기

"consistency"라는 말은 사실 두 개의 다른 요구를 뭉뚱그린다. 이를 분리하는 것이 이 절의 핵심 통찰이다.

Timeliness (적시성) Integrity (무결성)
의미 사용자가 최신 상태를 본다 데이터가 손상·모순 없음
위반 시 eventual consistency — 잠시 옛 값을 봄 corruption — 파생 데이터가 원본과 영구 불일치
복구 기다리면 해소(recoverable) 저절로 낫지 않음 — 수동 수정 필요
심층 설명 — timeliness vs integrity 구분: 핵심 주장은 대부분의 애플리케이션은 timeliness보다 integrity를 훨씬 더 중시한다는 것이다. 잔고가 몇 초 늦게 갱신되는 것(timeliness 위반)은 견딜 수 있지만, 돈이 사라지거나 이중 청구되는 것(integrity 위반)은 재앙이고 저절로 낫지 않는다. 그리고 ACID의 atomicity와 durability가 바로 integrity를 지키는 도구다. log 기반 dataflow는 이 통찰을 활용한다 — 이벤트를 원자적으로 로그에 append하고 결정론적· idempotent하게 파생하면, timeliness는 느슨하게(async, eventual) 두더라도 integrity는 완전히 보장할 수 있다. 즉 distributed transaction의 강한 동기화 없이도 무결성을 얻는다.

Log 기반 제약(constraint) 강제: uniqueness constraint(username 유일성, 잔고 ≥ 0, 재고 초과 판매 방지)는 consensus와 동등한 어려운 문제다(9장). 로그 기반 접근은 —

  • 같은 파티션으로 라우팅: 충돌 가능한 연산(같은 username을 claim하는 요청들)을 같은 파티션의 같은 로그로 보내면, 그 안의 total order로 "처음 claim한 자 승리, 나머지 거부"를 결정론적·linearizable하게 판정한다.
  • Multi-partition 비동기 버전: 여러 파티션에 걸친 제약(발신·수신 계좌가 다른 파티션)은 각 단계를 별도 이벤트로 쪼개 각각 idempotent하게 처리하면 원자적 커밋 없이 무결성을 유지한다 — 일시적 위반을 거칠 수 있고, 필요하면 보상 트랜잭션(compensating transaction)으로 되돌린다.

Coping with faults: 무결성은 어느 한 계층의 트랜잭션만으로 완성되지 않는다. 하드웨어·소프트웨어 버그, 중복 요청, 사람의 실수까지 감안하면 정확성은 end-to-end 식별자 + idempotence로 받쳐야 한다. log 기반 시스템은 결정론적 재처리가 가능하므로, 버그를 고친 뒤 로그를 다시 돌려 파생 데이터를 재생성하는 강력한 복구를 제공한다.

Trust, but Verify — 믿되, 검증하라

  • 맹신하지 말라: 하드웨어는 조용히 데이터를 손상(silent corruption)시키고 소프트웨어엔 버그가 있으며 복제·백업조차 완벽하지 않다. "데이터는 한 번 맞으면 영원히 맞다"는 가정은 틀렸다 — 시간이 지나면 어디선가 반드시 썩는다.
  • Auditing과 self-validation: 체크섬으로 손상을 탐지하고, 파생 데이터를 원본으로부터 재계산해 주기적으로 대조하며, 이벤트 로그의 불변성을 이용해 "현재 상태가 정말 이 이벤트들의 결과인가"를 확인한다(회계 감사처럼).
  • event-based의 감사 이점: 불변 이벤트 로그는 상태 변화의 완전한 이력(provenance)을 남겨, 문제 원인 추적·재구성이 mutable 상태만 있는 시스템보다 훨씬 쉽다. 정확성은 한 번 확보하고 끝이 아니라 계속 감사해 지켜야 한다.

Doing the Right Thing — 엔지니어의 윤리

기술을 넘어, 저자는 데이터 시스템을 만드는 사람의 윤리적 책임을 정면으로 다룬다. 코드와 데이터는 중립적이지 않다.

  • 편향·차별(bias & discrimination): 알고리즘이 대출·채용·보험·형량을 좌우할 때 학습 데이터에 담긴 과거의 편향이 그대로 재생산·증폭된다. "데이터는 객관적"이라는 믿음이 오히려 차별을 정당화한다.
  • Feedback loops: 예측이 현실을 바꾸고 그 현실이 다시 예측을 강화하는 악순환. 특정 동네를 "위험"으로 예측 → 경찰 집중 → 더 많은 체포 기록 → 더 "위험"한 데이터. 모델이 자기실현적 예언이 된다.
  • Algorithmic prisons: 알고리즘 판정으로 기회(신용·취업· 이동)가 막히면, 당사자는 왜 거부됐는지 알 수도 항변할 수도 없는 보이지 않는 감옥에 갇힌다.
  • Privacy와 consent: 형식적 약관 동의는 데이터가 어떻게 결합·추론·판매될지 이해한 informed consent가 아니다. 감시(surveillance)가 서비스의 대가로 정상화된다.
  • 데이터는 자산이자 위험물: "data is the new oil"은 데이터를 무한 축적할 자산으로만 본다. 하지만 축적된 개인정보는 유출·오용· 감시의 위험물(toxic asset)이기도 하다 — 원유가 환경을 오염시키듯. 모으지 않은 데이터는 유출될 수 없다.
  • 엔지니어의 책임: "나는 시키는 대로 만들 뿐"이라는 태도를 저자는 거부한다. 무엇을 수집하고 어떻게 쓰며 누구에게 해를 끼칠 수 있는지 설계 단계에서 능동적으로 고민하는 것이 직업적 책임이다 — 기술적 탁월함과 윤리적 책임은 분리될 수 없다.

쉬운 예시 / 비유

Data integration — 주방의 여러 도구

만능 조리기구 하나로 모든 요리를 할 수 없어 칼·냄비·오븐·믹서를 함께 쓴다. 중요한 건 같은 재료(원본)를 여러 도구가 각자 다루되 한 접시(일관된 결과)로 모이게 하는 조율이다.

System of record vs derived — 원본 계약서와 복사본

금고 속 원본 계약서 하나가 진실이고, 곳곳의 복사본·요약본·색인은 원본에서 언제든 다시 만들 수 있다. 복사본이 타 버려도 원본만 있으면 복원된다 — derived data의 본질.

로그의 total order — 은행 창구의 단일 대기줄

여러 창구(파생 시스템)가 각자 손님을 받으면 순번이 엉킨다. 하나의 번호표 줄에 세우면 모든 창구가 같은 순서로 처리해 결과가 어긋나지 않는다 — 이벤트 로그가 이 번호표 줄이다.

Unbundling database — 전문병원 네트워크

모든 과를 한 건물에 넣은 종합병원(monolithic DB) 대신, 전문병원들을 공통 진료기록(로그)으로 연결한 네트워크. 각 병원은 자기 분야에 최적화되고, 진료기록이 흐르며 전체가 일관된다.

Dataflow / reactive — 스프레드시트의 자동 재계산

A1 셀을 고치면 =A1*2인 B1이 저절로 갱신된다. 누구도 B1을 수동으로 다시 계산하지 않는다. 데이터 시스템도 원본이 바뀌면 파생 뷰가 저절로 갱신되게 짜자는 것이 dataflow 설계.

읽기 = materialized view — 미리 만든 반찬 vs 즉시 조리

쓰기(events)는 장을 보는 것, 읽기는 밥상을 차리는 것. 자주 나가는 반찬은 미리 만들어 두고(materialized view), 드문 요청은 주문받고 그때 조리(on-the-fly)한다. 어디까지 미리 만들지가 write path/read path의 균형이다.

End-to-end argument — 등기우편의 수령 확인

우체국(TCP/DB)이 "배달했다"고 보증해도, 받는 사람 본인의 서명(end-to-end 식별자)이 없으면 "정말 그 사람이 받았는가"는 보장되지 않는다. 편지가 두 번 와도 같은 등기번호를 보고 "이미 받은 것"으로 처리해야 중복이 없다.

Idempotence — 엘리베이터 버튼

버튼을 세 번 눌러도 엘리베이터는 한 번만 온다. 요청(누름)에 붙은 정체성(같은 층 호출)이 같으면 여러 번 처리해도 결과가 한 번과 같다 — idempotence.

Timeliness vs Integrity — 늦은 영수증 vs 틀린 잔고

카드를 긁고 영수증이 5초 늦게 뜨는 것(timeliness 위반)은 참을 수 있다. 하지만 결제 금액이 틀리거나 이중 청구되는 것(integrity 위반)은 절대 안 된다 — 저절로 낫지 않고 손으로 바로잡아야 한다.

Trust but verify — 정기 재고 실사

장부(파생 데이터)가 맞다고 믿지 말고 주기적으로 창고를 직접 세어(원본 재계산) 장부와 대조한다. 시간이 지나면 어디선가 수량이 어긋나기 마련 — 감사만이 무결성을 지킨다.

Data as hazard — 유조선의 기름

기름(데이터)은 자산이지만 유출되면 해안을 뒤덮는 재앙이다. "새로운 석유"라 무작정 채굴·비축하기 전에, 그 저장고가 터졌을 때의 피해(유출· 감시·오용)를 먼저 생각해야 한다. 모으지 않은 기름은 새지 않는다.

핵심 용어 정리 (Glossary)

용어 정의
Data integration 이질적 여러 저장소·인덱스·처리 시스템을 하나의 일관된 시스템으로 통합하는 문제
System of record 새 데이터가 처음 권위 있게 기록되는 원본(source of truth)
Derived data 원본을 변환해 만든 중복(index, cache, view, ML model) — 잃어도 재생성 가능
Dataflow 원본의 변화를 batch·stream 처리로 파생 데이터에 전파하는 흐름 중심 아키텍처
Total order (of events) 모든 쓰기를 하나의 로그 순서로 묶어 같은 순서로 소비 — 충돌 방지
Log-based derivation distributed transaction 대신 로그 + idempotence로 파생 데이터를 느슨하게 동기화
Unbundling databases DB 내부 기능(index, materialized view, replication log)을 로그를 축으로 분해·재조립
Federated / Unbundled database 통합된 읽기(unified read, polystore) / 로그로 쓰기 통합(unified write) 후 파생
Derivation function 원본에서 파생 데이터를 만드는 앱 특화 함수 (stream processor가 실행)
Materialized view 미리 계산해 저장한 읽기용 뷰 — read를 살아있는 뷰로 보는 관점
Write path / Read path 쓰기 시 미리 계산 vs 읽기 시 계산 — 어디까지 선계산할지의 트레이드오프
End-to-end argument 어떤 정확성 보장은 통신 경로의 양 끝점에서만 완전 — 하위 계층만으론 불충분 (Saltzer, Reed, Clark)
Exactly-once (effectively-once) 결과가 한 번 처리한 것과 같아지게 함 — idempotence + dedup으로 달성
Idempotence 같은 연산을 여러 번 적용해도 결과가 한 번과 동일
Unique operation identifier 요청을 유일 식별하는, end-to-end로 전달되는 ID — dedup의 열쇠
Timeliness 최신 상태를 보는 성질 — 위반은 eventual consistency(복구 가능)
Integrity 데이터가 손상·모순 없는 성질 — 위반은 corruption(복구 어려움)
Constraint enforcement (log) 충돌 연산을 같은 파티션 로그로 라우팅해 결정론적으로 판정
Compensating transaction 잘못 적용된 연산을 되돌리는 보상 연산 (다단계 비동기 제약에서)
Auditing / Silent corruption 파생 데이터를 원본과 대조·재계산해 지속 검증 / 조용히 일어나는 데이터 손상
Provenance 불변 이벤트 로그가 남기는 상태 변화의 완전한 이력 (감사 이점)
Feedback loop 예측이 현실을 바꾸고 그 현실이 예측을 강화하는 자기실현적 악순환
Algorithmic prison 알고리즘 판정으로 기회가 막히나 이유를 알거나 항변할 수 없는 상태
Data as asset and hazard 데이터는 자산이자 유출·감시의 위험물(toxic asset) — "data is the new oil"의 이면
Informed consent 데이터가 어떻게 쓰일지 진정으로 이해한 동의 — 형식적 약관 동의와 구별

한 줄 요약

만능 도구는 없으므로 미래의 데이터 시스템은 system of record와 derived data를 구분하고, 로그의 total order + idempotence로 파생 데이터를 느슨하게 동기화하는 unbundled database(dataflow 중심 설계)로 나아간다 — distributed transaction 대신 로그 기반 파생을, 정확성은 어느 한 계층이 아니라 end-to-end 식별자로, 일관성은 timeliness(복구 가능)와 integrity(치명적)의 구분으로 다루며, 무엇보다 데이터를 믿되 끊임없이 검증(auditing)하고, 편향·감시·프라이버시에 대한 엔지니어의 윤리적 책임을 잊지 말아야 한다.
