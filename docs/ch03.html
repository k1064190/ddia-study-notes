Ch3. Storage and Retrieval

DB 엔진이 데이터를 디스크에 실제로 어떻게 쌓고, 어떻게 빨리 찾아내는가 — LSM-tree vs B-tree, 그리고 분석을 위한 column-oriented storage.

챕터 개요

데이터베이스는 가장 근본적으로 두 가지 일을 한다 — 데이터를 저장하고, 나중에 다시 꺼내오는 것. 2장이 "데이터를 어떤 모델로 표현할까"였다면, 3장은 그 한 단계 아래, 즉 storage engine이 데이터를 디스크에 실제로 어떻게 쌓고 어떻게 빨리 찾아내는가를 다룬다.

앱 개발자가 직접 storage engine을 만들 일은 드물지만, 내 워크로드에 맞는 엔진을 고르고 튜닝하려면 내부 동작을 알아야 한다. 특히 트랜잭션 처리용(OLTP) 엔진과 분석용(OLAP) 엔진은 완전히 다르게 최적화되어 있다.

이 챕터의 큰 두 축:

  • OLTP 스토리지 엔진의 두 계열 — log-structured(LSM-tree) vs page-oriented(B-tree)
  • 분석(OLAP)을 위한 저장 — data warehouse, star schema, column-oriented storage

핵심 개념 정리

세상에서 가장 단순한 데이터베이스

key-value를 파일에 append하고, 읽을 때 파일 전체를 훑어 마지막 값을 찾는 방식을 생각해 보자.

  • 쓰기: 파일 끝에 한 줄 추가 → 매우 빠름 (O(1), append-only).
  • 읽기: 파일 전체 스캔 → 레코드 수 n에 비례 (O(n)) → 데이터가 커지면 끔찍하게 느려진다.

여기서 핵심 개념이 등장한다: index(인덱스). 인덱스는 원본 데이터에서 파생된 추가 자료구조로, 원하는 데이터의 위치를 빨리 찾게 해준다.

  • 읽기는 빨라지지만, 쓰기는 느려진다 — 쓸 때마다 인덱스도 갱신해야 하기 때문.
  • 그래서 인덱스는 자동 생성되지 않고 개발자가 워크로드를 보고 선택한다. 이것이 storage의 가장 중요한 trade-off다.

Hash Index와 Bitcask

가장 단순한 인덱스는 in-memory hash map이다. key → 파일 안의 byte offset을 기억한다.

  • 쓰기: 파일에 append하고 hash map의 offset을 갱신.
  • 읽기: hash map에서 offset을 찾아 디스크에서 한 번에 읽음.
  • 실제 구현 예: Bitcask(Riak의 기본 엔진). key가 전부 메모리에 들어가면 값이 디스크에 있어도 매우 빠르다.

append-only 로그가 무한히 커지는 문제 → compaction으로 해결:

  • 로그를 일정 크기의 segment로 나눈다.
  • compaction: 각 segment에서 같은 key의 중복을 버리고 최신 값만 남긴다.
  • 여러 segment를 백그라운드에서 merge하면서 동시에 compaction → 디스크 사용량 절감.

실무 디테일:

  • 파일 포맷: 텍스트보다 binary가 효율적.
  • 삭제: tombstone(삭제 표식) 레코드를 남겨두고 merge 때 실제 제거.
  • crash recovery: hash map snapshot을 디스크에 저장해 재시작 시 빠르게 복구.
  • 부분 기록(partial write): checksum으로 깨진 레코드 감지.
  • 동시성: 보통 단일 writer thread(append는 순차여야 하므로), 읽기는 여러 thread 가능.

append-only가 좋은 이유: sequential write는 random write보다 (HDD/SSD 모두) 훨씬 빠르고, 동시성·crash recovery가 단순해진다.

hash index의 한계:

  1. hash table이 메모리에 다 들어가야 한다 (key가 너무 많으면 곤란).
  2. range query(범위 질의)가 비효율적 — key1~key2 스캔을 한 번에 못 한다.

SSTable과 LSM-Tree

SSTable (Sorted String Table): segment 파일 안의 key-value 쌍을 key 기준으로 정렬해 두고, 각 key는 한 segment에 한 번만 등장하게 한다. 이 "정렬" 하나로 큰 이점이 생긴다:

  1. merge가 효율적: 정렬된 여러 파일을 mergesort처럼 합치므로 메모리보다 큰 파일도 합칠 수 있다.
  2. sparse index: 모든 key를 메모리에 둘 필요 없이 몇 KB마다 하나씩만 인덱스에 두면 된다 — 정렬돼 있으니 그 사이만 스캔하면 찾을 수 있다.
  3. block 압축: 인덱스 항목 사이 구간을 묶어 압축 후 저장 → 디스크/대역폭 절약.

어떻게 정렬 상태를 유지하나? (memtable 방식)

  • 쓰기는 일단 in-memory balanced tree(red-black tree, AVL 등)인 memtable에 넣는다.
  • memtable이 임계치를 넘으면 정렬된 상태 그대로 SSTable 파일로 디스크에 기록한다.
  • 읽기: memtable → 가장 최신 SSTable → 더 오래된 SSTable 순으로 탐색.
  • 백그라운드에서 주기적으로 merge & compaction.
  • crash 대비: memtable은 휘발성이므로 별도의 append-only log(WAL)에 먼저 기록해 두고, SSTable로 내려쓰면 그 log를 버린다.

이 구조 전체를 LSM-Tree (Log-Structured Merge-Tree)라 한다. 사용처: LevelDB, RocksDB, Cassandra, HBase, 그리고 검색엔진 Lucene(용어 사전).

성능 최적화:

  • Bloom filter: "이 key가 존재하지 않는다"를 메모리에서 빠르게 판단 → 없는 key를 찾느라 여러 SSTable을 디스크에서 뒤지는 낭비를 막는다. false positive는 있어도 false negative는 없다.
  • compaction 전략:
    • size-tiered: 작은 SSTable을 점점 큰 것으로 합침 (HBase, Cassandra).
    • leveled: key 범위를 레벨별로 나눠 더 잘게 점진적으로 합침 → 디스크 효율↑ (LevelDB, RocksDB).
심층 설명 — LSM의 읽기/쓰기 경로와 sequential write가 이기는 이유. 메커니즘: 쓰기는 WAL append(순차) + memtable 삽입(메모리)뿐이므로 디스크 관점에서는 항상 "파일 끝에 이어 쓰기"다. HDD는 헤드 이동(seek)이 없어지고, SSD는 페이지 단위 erase-before-write 특성상 흩어진 덮어쓰기보다 큰 순차 블록 쓰기가 훨씬 효율적이다. 반면 읽기는 memtable → 최신 SSTable → 오래된 SSTable 순으로 여러 곳을 확인해야 할 수 있어 상대적으로 비싸며, 이 비용을 Bloom filter(부재 판단)와 compaction(segment 수 줄이기)이 깎아준다. 실무 함의: 쓰기 폭주 워크로드(로그, 이벤트 수집)에 LSM 계열(RocksDB, Cassandra)이 흔히 선택되는 이유가 이것이다. 흔한 오해: "LSM은 디스크에 한 번만 쓴다"가 아니다 — WAL, flush, 반복되는 compaction으로 데이터는 여러 번 다시 쓰인다. 다만 그 모든 쓰기가 순차라서 총비용이 낮은 것이다.

B-Tree

가장 널리 쓰이는 인덱스. 거의 모든 관계형 DB의 표준이고 많은 비관계형 DB도 사용한다. SSTable처럼 key 기준 정렬을 유지하지만 설계 철학이 다르다.

  • DB를 고정 크기 block/page(전통적으로 4KB)로 나누고, 읽기/쓰기를 한 page 단위로 한다.
  • page는 다른 page를 pointer(디스크 주소)로 가리킨다 → page들의 트리.
  • root page → branching pages → leaf page 구조. leaf에 실제 값(또는 값의 위치)이 있다.
  • branching factor: 한 page가 가리키는 자식 수 (보통 수백). depth는 O(log n). 예: 4KB page, branching factor 500이면 4단계 트리로 256TB까지 저장 가능.

갱신/삽입:

  • 갱신: leaf까지 내려가 값을 바꾸고 그 page를 제자리에 덮어쓴다(overwrite in place).
  • 삽입: 들어갈 page를 찾아 추가. 자리가 없으면 page를 둘로 split하고 부모를 갱신 → 트리가 항상 balanced하게 유지된다 (depth O(log n)).

신뢰성 확보:

  • B-tree는 LSM과 달리 제자리 덮어쓰기라, 도중에 crash가 나면 인덱스가 깨질 수 있다.
  • → WAL (Write-Ahead Log, redo log): 모든 변경을 먼저 append-only log에 기록한 뒤 page를 수정. crash 후 이 log로 복구한다.
  • 동시성: 여러 thread가 동시에 page를 만지면 위험 → latch(가벼운 lock)로 보호.

최적화: copy-on-write(WAL 대신, 동시성에도 유리), key 축약(B+ tree), leaf page를 디스크에 순차 배치, leaf 간 형제 pointer 추가 등.

LSM-Tree vs B-Tree 비교

항목 LSM-Tree B-Tree
쓰기 일반적으로 빠름 (순차 쓰기) 상대적으로 느림 (제자리 쓰기 + split)
읽기 상대적으로 느릴 수 있음 (여러 segment 확인) 일반적으로 빠름 (한 위치)
write amplification 낮은 편 높은 편 (WAL + page 전체, split)
압축/디스크 사용 더 잘 압축, 단편화 적음 page 단위라 빈 공간(fragmentation) 발생
단점 compaction이 읽기/쓰기와 자원 경쟁 → tail latency(p99) 튐, key가 여러 segment에 존재 각 key가 정확히 한 곳에 존재 → 트랜잭션·lock에 유리

Write amplification(쓰기 증폭): 한 번의 논리적 쓰기가 실제로는 디스크에 여러 번 쓰이는 현상. SSD 수명과 throughput에 영향을 준다. LSM이 보통 더 유리하지만, compaction이 들어오는 쓰기를 못 따라가면 디스크가 차고 읽기가 느려질 수 있다.

요약: 쓰기 많으면 LSM, 읽기 위주·트랜잭션 중요하면 B-tree가 흔한 출발점. 단, 실제 성능은 워크로드로 측정해야 한다.

심층 설명 — B-tree vs LSM trade-off와 compaction으로 인한 tail latency. 메커니즘: B-tree의 쓰기 비용은 예측 가능하다 — WAL 한 번 + page 덮어쓰기(가끔 split). LSM의 쓰기는 평균적으로 더 싸지만, 백그라운드 compaction이 디스크 대역폭과 CPU를 소비하므로 그 순간 들어온 쿼리의 지연이 튀어 오른다. 평균(p50) latency는 좋아 보여도 p99/p999가 불안정해지는 것이 LSM의 대표적 약점이다. 또한 compaction이 유입 쓰기 속도를 못 따라가면 미병합 segment가 쌓여 읽기가 점점 느려지고 디스크가 차오르는 악순환이 생긴다. 실무 함의: latency SLO가 엄격한 서비스라면 평균이 아니라 tail을 벤치마크해야 하고, LSM 엔진에서는 compaction 스로틀링/스케줄링 튜닝이 핵심 운영 포인트다. 흔한 오해: "LSM이 무조건 빠르다"는 틀렸다 — 각 key가 정확히 한 곳에만 있는 B-tree는 트랜잭션 lock 구현과 예측 가능한 읽기에서 여전히 강하다.

기타 인덱스 구조

  • Secondary index(보조 인덱스): primary key 외의 컬럼에 거는 인덱스. 같은 값을 가진 row가 여럿이므로 key 중복을 허용한다.
  • 인덱스에 값을 어떻게 둘까?
    • heap file: 인덱스는 row의 위치만 가리키고, 실제 row는 별도 heap에 저장 (중복 없음).
    • clustered index: 인덱스 안에 row 전체를 같이 저장 (예: MySQL InnoDB의 primary key). 읽기는 빠르지만 쓰기·저장 부담↑.
    • covering index: 일부 컬럼만 인덱스에 포함 → 그 쿼리는 heap을 안 봐도 된다 (clustered와 nonclustered의 중간).
  • multi-column index:
    • concatenated index: 여러 컬럼을 이어붙여 하나의 key로 (예: (lastname, firstname)). 순서가 중요 — 앞 컬럼 없이 뒷 컬럼만으로는 못 쓴다.
    • multi-dimensional index: 위경도 같은 다차원 검색에는 R-tree 등 (지리 공간 데이터).
  • full-text search / fuzzy index: 오타·유의어까지 검색. Lucene은 Levenshtein automaton으로 편집 거리 내 단어를 찾는다.
  • in-memory database: 데이터를 RAM에 둔다 (Redis, Memcached, VoltDB, MemSQL, Oracle TimesTen).
    • durability: log/snapshot/replication으로 보강.
    • 빠른 진짜 이유: "디스크를 안 읽어서"가 아니라 디스크용으로 인코딩하는 오버헤드를 피해서다.
    • anti-caching: 메모리 부족 시 잘 안 쓰는 데이터를 디스크로 내린다.

OLTP vs OLAP

특성 OLTP (Online Transaction Processing) OLAP (Online Analytic Processing)
읽기 패턴 key로 소수 레코드 조회 대량 레코드를 집계(aggregate)
쓰기 패턴 사용자 입력 기반 random-access, 저지연 bulk import(ETL) 또는 event stream
사용자 end user / web app analyst / 의사결정
데이터 내용 최신 상태(current state) 이벤트의 역사(history)
데이터 크기 GB ~ TB TB ~ PB

같은 DB로 둘 다 하면 분석 쿼리가 OLTP 성능을 망친다 → 분석용은 따로 둔다 (data warehouse).

Data Warehousing — ETL, Star & Snowflake Schema

  • OLTP 시스템들의 데이터를 ETL(Extract–Transform–Load)로 추출·변환해 별도의 분석 전용 DB(data warehouse)에 적재한다. 분석가가 마음껏 무거운 쿼리를 돌려도 운영 DB에 영향이 없다.
  • Star schema(dimensional modeling):
    • 중앙에 fact table — 이벤트 1건 = 1 row (예: 판매 1건). 보통 매우 크다(수십억 row).
    • 주변에 dimension table들 — who/what/where/when/how/why (제품·고객·매장·날짜 등).
    • fact table이 dimension table들을 foreign key로 가리키는 모양이 "별"처럼 보인다.
  • Snowflake schema: dimension table을 더 정규화(세분화)한 변형. star보다 정규화 정도가 높지만 다루기는 더 복잡하다.

Column-Oriented Storage (열 지향 저장)

data warehouse의 fact table은 컬럼이 수백 개인데, 한 분석 쿼리는 보통 그중 몇 개 컬럼만 쓰면서 수십억 row를 훑는다. 여기서 저장 방식이 결정적이다.

  • Row-oriented(행 지향): 한 row의 모든 컬럼 값을 디스크에 붙여서 저장 (OLTP의 기본). → 컬럼 몇 개만 필요해도 row 전체를 읽어야 함 → 분석엔 낭비.
  • Column-oriented(열 지향): 각 컬럼의 값들을 따로 모아서 저장. → 쿼리에 쓰이는 컬럼 파일만 읽으면 됨 → I/O 대폭 절감. (BigQuery, Vertica, Parquet 등)

column 압축:

  • bitmap encoding: 컬럼의 distinct 값마다 비트맵을 만든다. cardinality(고유값 수)가 낮으면 매우 효율적. WHERE x IN (...) 같은 조건은 비트맵 OR/AND로 초고속 처리.
  • run-length encoding: 같은 값이 연속되면 "값 × 개수"로 압축.
  • 추가 효과: 압축된 데이터는 CPU 캐시에 잘 들어가 vectorized processing(여러 값 한꺼번에 연산)에 유리 → 메모리 대역폭 병목 완화.

sort order(정렬 순서):

  • 컬럼 저장에서도 row들을 특정 컬럼 기준으로 정렬해 둘 수 있다 — 단, 모든 컬럼을 같은 순서로!
  • 자주 거르는 컬럼으로 정렬하면 range 스캔이 빨라지고, 무엇보다 정렬된 컬럼은 run-length encoding이 극도로 잘 먹힌다 (같은 값이 길게 연속).
  • C-Store/Vertica는 같은 데이터를 여러 정렬 순서로 복제본마다 다르게 저장해, 쿼리에 맞는 정렬본을 골라 쓴다.

column 저장에 쓰기: 정렬·압축된 컬럼은 B-tree식 제자리 삽입이 어렵다 → LSM-tree 방식 사용: 쓰기를 in-memory store에 모았다가 컬럼 파일과 merge해서 디스크로 내린다 (Vertica가 이렇게 한다).

집계 가속:

  • materialized view: 쿼리 결과를 미리 계산해 저장 (실제 테이블처럼 디스크에 둠).
  • data cube / OLAP cube: 여러 dimension 조합의 집계를 격자(큐브)로 미리 계산 → 특정 집계는 즉시 조회. 단, 원자료 쿼리만큼 자유롭지는 않다.
심층 설명 — column storage + 저 cardinality 압축이 왜 그렇게 잘 통하나. 메커니즘: 같은 컬럼의 값들은 타입과 분포가 균질하다(예: "국가" 컬럼은 200여 개 값만 반복). distinct 값이 n개면 bitmap n개로 컬럼 전체를 표현할 수 있고, 정렬까지 해두면 같은 값이 수백만 개씩 연속되어 run-length encoding이 비트맵을 몇 바이트로 줄인다. 그 결과 수십억 row의 컬럼이 CPU 캐시에 들어갈 만큼 작아지고, AND/OR 필터는 비트 연산 몇 번으로 끝난다 — 디스크 I/O뿐 아니라 메모리 대역폭 병목까지 풀리는 것이다. 실무 함의: Parquet 같은 포맷에서 파티셔닝/정렬 키를 저 cardinality 필터 컬럼으로 잡으면 스캔 비용이 극적으로 줄어든다. 흔한 오해: column storage가 "항상" 좋은 게 아니다 — 한 row 전체를 읽고 쓰는 OLTP 패턴에서는 컬럼들을 다시 조립해야 해서 오히려 손해다.

쉬운 예시 / 비유

Index의 trade-off = 책 뒤의 찾아보기(색인)
색인이 있으면 단어를 빨리 찾지만, 책 내용을 고칠 때마다 색인도 다시 손봐야 한다. 읽기↑, 쓰기↓ — 공짜가 아니다.
Append-only log + compaction = 거래 장부
매번 새 줄로 적기만 하는 장부. "철수 잔액 = 5만원"을 여러 번 고쳐 적었다면, 정리(compaction) 때 가장 마지막 줄만 남기고 옛 줄은 버린다.
SSTable의 sparse index = 영어사전 찾기
정렬된 사전에서 "rabbit"을 찾을 때, 페이지마다 첫 단어만 적힌 색인으로 대략 위치를 잡고 그 페이지 안만 훑는다. 모든 단어의 위치를 외울 필요가 없다.
Bloom filter = 도서관 사서
"이 책 여기 있어요?"라고 물으면 사서가 목록만 슬쩍 보고 "확실히 없어요"는 즉답하고, "있을 수도 있어요"면 그때 서가를 뒤진다. 없는 걸 찾느라 서가(디스크)를 헛되이 뒤지지 않게 해준다.
B-tree의 page split = 꽉 찬 바인더
새 종이를 끼울 자리가 없으면 바인더를 둘로 나누고, 목차(부모 page)에 "이제 바인더가 두 개"라고 갱신한다.
Row vs Column 저장 = 전교생 수학 평균
학생 100만 명의 성적표에서 수학 평균을 구할 때, row 저장은 학생마다 전 과목 성적표를 다 꺼내야 하고(낭비), column 저장은 "수학 점수" 묶음 하나만 꺼내면 된다.

한 줄 요약

Storage 엔진은 워크로드에 따라 갈린다 — 트랜잭션(OLTP)은 쓰기에 강한 LSM-tree나 읽기에 강한 B-tree로, 분석(OLAP)은 필요한 컬럼만 읽는 column-oriented storage로 최적화하며, 모든 인덱스는 "읽기는 빠르게, 쓰기는 느리게"라는 trade-off 위에 있다.
