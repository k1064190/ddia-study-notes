Ch8. The Trouble with Distributed Systems

잘못될 수 있는 것은 반드시 잘못된다 — 신뢰할 수 없는 네트워크·시계· 프로세스 위에서 partial failure와 함께 사는 법, 그리고 quorum·fencing token·system model로 "진실"을 정의하는 법

챕터 개요

단일 머신의 소프트웨어는 결정적(deterministic)이다 — 하드웨어가 멀쩡하면 같은 연산은 항상 같은 결과를 내고, 고장 나면 시스템 전체가 죽는다(전부 아니면 전무). 분산 시스템의 정의적 특징은 partial failure(부분 실패) — 일부는 망가졌는데 나머지는 멀쩡히 동작하는 상태 — 이며, 게다가 부분 실패는 비결정적(nondeterministic)이다. 뭔가가 성공했는지조차 알 수 없을 때가 있다.

믿을 수 없는 것은 세 가지다: 네트워크(패킷은 유실·지연되고 응답이 없으면 원인을 알 수 없다), 시계(노드마다 다르게 흐르고 툭하면 튄다), 프로세스 자체(GC 등으로 언제든 멈춘다). 결론은 역설적이다 — 신뢰할 수 없는 부품들로 신뢰할 수 있는 시스템을 짓는 것이 분산 시스템 엔지니어링의 본질이다.

Supercomputer(HPC) vs Cloud — HPC는 한 노드가 죽으면 전체 작업을 멈추고 checkpoint에서 재시작한다: 부분 실패를 전체 실패로 승격시켜 단일 머신처럼 다룬다. cloud computing은 상용(commodity) 하드웨어 위에서 온라인 서비스를 멈출 수 없으므로, fault를 전제하고 소프트웨어로 fault tolerance를 구축한다 — "고장은 언제나 어딘가에서 일어나고 있다". TCP가 신뢰성 낮은 IP 위에 쌓이듯, 신뢰성 낮은 층 위에 더 신뢰성 높은 층을 쌓는다.

핵심 개념 정리

Unreliable Networks — 신뢰할 수 없는 네트워크

인터넷과 데이터센터 네트워크는 asynchronous packet network다 — 패킷 전달 시간에 아무 보장이 없다. 요청을 보내고 응답을 기다릴 때 잘못될 수 있는 경우:

# 무엇이 잘못되나 예
1 요청이 유실됨 케이블 뽑힘
2 요청이 queue에서 대기 중 수신자 과부하 — 나중에 전달됨
3 원격 노드가 crash했음 전원 차단
4 원격 노드가 일시 정지 후 재개 GC pause — 죽은 게 아니다
5 처리됐는데 응답이 유실됨 스위치 설정 오류
6 응답이 지연됨 네트워크·수신자 과부하

핵심: 응답을 받지 못하면 이 경우들을 구별할 수 없다 — 요청이 갔는지, 처리됐는지조차 모른다. 유일한 정보는 "아직 응답이 없다"뿐이고, 실무적 대응은 timeout이다. 하지만 timeout이 나도 상대가 요청을 처리했는지는 여전히 모른다 — 그저 포기하는 것이다.

  • Network faults in practice: 한 중견 데이터센터 연구에서 월 ~12건의 네트워크 장애; EC2는 transient glitch로 유명; 상어가 해저 케이블을 물어뜯고; 스위치 설정 오류로 한 방향으로만 패킷이 흐르기도 한다. 네트워크 일부가 단절되는 것이 network partition(netsplit). 장애는 없앨 수 없으므로 소프트웨어의 대응을 Chaos Monkey류로 테스트해야 한다.
  • Detecting faults: 일부 피드백은 있다 — 프로세스가 죽고 OS는 살아 있으면 RST/FIN으로 연결 거부; crash를 스크립트가 다른 노드에 통지(HBase); 스위치 관리 인터페이스의 링크 다운 감지; ICMP Destination Unreachable. 그러나 모두 있을 수도 없을 수도 있는 부가 정보 — 처리 확신은 애플리케이션의 긍정 응답뿐이고, 결국 timeout에 의존한다.
  • Sync vs async network: 전화망(circuit-switched)은 경로 전체에 대역폭을 예약해 큐잉이 없고 bounded delay를 보장한다. TCP/이더넷은 packet-switched — 데이터가 bursty하므로 동적으로 대역폭을 나눠 링크 활용률을 극대화한다. 결국 latency 보장 vs 자원 활용률(비용)의 trade-off이며, 현재 기술로는 큐잉 지연의 상한을 보장할 수 없다.
심층 설명 — timeout 딜레마와 queueing delay의 원인들: timeout이 너무 길면 죽은 노드 선언까지 사용자가 에러를 계속 보고, 너무 짧으면 살아 있는(느린) 노드를 성급하게 사망 선고한다 — 그 노드가 하던 작업(이메일 발송 등)을 다른 노드가 또 하면 중복 수행이 되고, 느림의 원인이 과부하였다면 부하 이전이 연쇄 과부하를 낳아 cascading failure(모든 노드가 서로를 죽었다고 선언)로 번진다. 이상적 공식 timeout = 2d + r은 현실에서 무너진다 — 지연 d도 처리 시간 r도 unbounded이기 때문. 변동의 주범은 여러 층의 queueing이다: ① 스위치 큐(여럿이 같은 목적지로 동시 전송 — 가득 차면 drop, 혼잡), ② 목적지 OS 큐(CPU가 바쁠 때), ③ VM 스케줄링 대기(VM monitor가 큐잉), ④ TCP flow control(송신자 스스로 자제), ⑤ TCP 재전송(유실 패킷의 timeout 대기가 애플리케이션에는 지연으로 보임). 멀티테넌트 클라우드에서는 noisy neighbor 때문에 특히 심하다. 그래서 timeout은 상수가 아니라 지연 분포를 관측해 실험적으로 정하거나, Phi Accrual failure detector(Akka, Cassandra)처럼 응답 시간 분포를 계속 측정해 자동 조정한다.

Unreliable Clocks — 신뢰할 수 없는 시계

Time-of-day clock (벽시계) Monotonic clock
반환값 현재 날짜·시간 (CLOCK_REALTIME, System.currentTimeMillis()) 절대값은 무의미 — 두 시점의 차이만 유효 (CLOCK_MONOTONIC, System.nanoTime())
NTP와의 관계 동기화 대상 — 뒤로 점프할 수 있다 진행 속도만 미세 조정(slew) — 점프 없음, 항상 전진
용도 날짜·시간 표기 (duration 측정 부적합) duration·timeout 측정에 사용해야 한다
  • 동기화의 현실: 수정 발진기 drift(Google 가정 200ppm — 하루 17초); NTP 정확도는 네트워크 지연에 제한(공용 인터넷 35ms+); 너무 벌어지면 강제 리셋(점프); 잘못된 NTP 서버·방화벽; leap second(59초/61초짜리 1분 — 실제 장애 다수, smearing으로 완화); VM의 가상화된 시계; 사용자 기기의 시계는 아예 신뢰 불가.
  • 시계 문제는 조용하다: 시계가 서서히 어긋나도 소프트웨어는 잘 도는 것처럼 보이고, 데이터가 소리 없이 유실된다 — 극적인 crash보다 위험하다.
  • Confidence interval과 Spanner TrueTime: 시계 읽기는 하나의 시점이 아니라 불확실성 구간으로 봐야 한다. Google Spanner의 TrueTime API는 [earliest, latest]를 반환하고, snapshot isolation의 트랜잭션 순서를 위해 commit 전에 구간 길이만큼 의도적으로 기다린다(waiting out the uncertainty). 대기를 짧게 하려고 데이터센터마다 GPS 수신기·원자시계를 두어 불확실성을 ~7ms로 유지한다.
심층 설명 — monotonic vs time-of-day clock과 LWW의 위험: timeout을 벽시계로 재면 NTP가 시계를 뒤로 돌리는 순간 "음수 경과 시간" 같은 괴현상이 생긴다 — 그래서 duration은 반드시 monotonic clock으로 잰다. 더 위험한 것은 노드 간 순서를 물리 시계에 맡기는 것이다. Cassandra류의 LWW(last write wins)는 timestamp가 가장 큰 쓰기를 승자로 삼는데, 노드 간 시계 skew가 있으면 — 시계가 느린 노드에서 인과적으로 나중에 일어난 쓰기가, 시계 빠른 노드의 이전 쓰기에게 에러 하나 없이 조용히 패배해 버려진다. causality 위반 + silent data loss. 시계를 아무리 잘 동기화해도(수십 ms 오차) 그보다 빠른 연속 쓰기의 순서는 뒤집힐 수 있다. 교훈: 사건의 순서에는 물리 시계가 아니라 logical clock(단조 증가 카운터 — 상대적 순서만 기록)을 써야 하고, 물리 시계를 꼭 써야 한다면 Spanner처럼 읽기를 confidence interval로 취급하고 불확실성을 기다려 없애야 한다.

Process Pauses — 프로세스는 언제든 멈춘다

lease 기반 leader 판정 코드를 생각하라 — "만료까지 10초 남았으니 안전"이라 판단하고 쓰기를 수행하는 순간 스레드가 15초 멈췄다면? 재개된 코드는 자신이 멈췄다는 사실조차 모른 채 이미 만료된 lease로 쓰기를 계속한다. 멈춤의 원인:

  • GC: "stop-the-world" — 때로 분 단위 정지.
  • VM suspend/resume(live migration 포함), 노트북 뚜껑 닫기 등 기기 suspend.
  • OS context switch, hypervisor의 CPU steal(다른 VM에게 CPU를 뺏김).
  • 예기치 못한 디스크 I/O(Java classloader의 lazy loading 등), swap/thrashing(서버에서 swap을 끄는 이유), SIGSTOP(Ctrl-Z — SIGCONT까지 프로세스는 아무것도 모른 채 정지).

단일 머신에는 mutex·semaphore가 있지만 분산에는 공유 메모리가 없다 — 노드는 통째로 멈출 수 있고 나머지는 계속 달린다. 응답 deadline이 보장되는 hard real-time 시스템(RTOS)이 항공기·자동차 제어에는 존재하지만, 언어·라이브러리 전면 제약과 막대한 비용, 낮은 처리량 때문에 서버 데이터 시스템에는 비경제적이다. 현실적 완화책: ① GC pause를 계획된 순단처럼 취급(런타임이 GC 임박을 경고하면 요청을 다른 노드로 우회), ② 단명 객체만 빠른 GC로 처리하고 full GC 전에 프로세스를 주기적으로 재시작(rolling restart).

Knowledge, Truth, and Lies — 진실은 다수결로 정의된다

  • 노드는 자기 판단을 믿을 수 없다: 요청은 다 처리하는데 응답만 안 나가는 반쪽 단절 노드는 자신이 멀쩡하다고 확신하지만 남들은 사망 선고를 내린다; 1분 GC로 멈춘 노드는 깨어나서도 1분이 지난 줄 모른다. 그래서 분산 알고리즘은 quorum(과반수) 투표에 의존한다 — 과반이 죽었다고 선언하면 (본인이 살아 있다고 느껴도) 죽은 것으로 간주된다. 과반을 요구하는 이유는 상충하는 두 결정이 동시에 성립할 수 없게 하기 위함.
  • Byzantine faults: 노드가 "거짓말"(임의의 오작동, 조작된 응답)을 할 수 있는 fault. 견디는 시스템이 Byzantine fault-tolerant(비잔틴 장군 문제에서 유래). 필요한 곳 — 항공우주(방사선에 의한 메모리 오염), 블록체인(중앙 권위 없는 상호 불신 참여자). 대부분의 데이터 시스템은 전 노드를 한 조직이 통제하므로 non-Byzantine을 가정한다 (BFT 프로토콜은 비싸고 복잡하다; 클라이언트의 악의는 서버의 입력 검증으로 처리). 다만 약한 거짓말 방어는 가치가 있다: 패킷 오염 대비 checksum(TCP checksum이 못 잡는 것도 있다), 입력 sanity check/validation, 여러 NTP 서버 참조로 outlier 제거.
심층 설명 — fencing token이 필요한 이유: lock/lease를 가진 클라이언트가 "오직 나만 쓸 수 있다"고 믿는 것 자체가 함정이다. GC pause 중 lease가 만료되고 다른 노드가 lock을 받았는데, 깨어난 옛 보유자는 자신이 멈췄다는 사실을 모른 채 쓰기를 계속한다 — HBase에서 실제로 데이터가 깨진 버그다. lock만으로 못 막는 이유는 검사 시점("나 아직 유효하지?")과 쓰기 시점 사이에 얼마든지 pause가 끼어들 수 있기 때문. 해법은 신뢰의 방향을 뒤집는 것이다 — lock 서버가 lock을 줄 때마다 단조 증가하는 fencing token을 발급하고, storage 쪽이 쓰기마다 토큰을 검사해 이미 더 높은 토큰을 처리했다면 낮은 토큰의 쓰기를 거부한다(ZooKeeper라면 zxid/cversion을 토큰으로 쓸 수 있다). 낡은 lease로 깨어난 클라이언트의 쓰기는 오래된 토큰이라 자동 차단된다. 교훈: 클라이언트가 스스로 착하게 굴기를 기대하는 시스템은 안전하지 않다 — 검사는 서버 측에서 해야 한다. (단, fencing token도 "몰라서" 틀리는 노드까지만 막는다 — 토큰을 위조하는 거짓말쟁이는 Byzantine fault의 영역이다.)

System Models — 가정을 형식화하기

축 모델 가정
Timing Synchronous 네트워크 지연·pause·시계 오차 모두 bounded — 현실 데이터 시스템에는 비현실적
Partially synchronous 대부분의 시간엔 synchronous, 가끔 한도 초과 — 가장 현실적
Asynchronous 타이밍 가정 전무(시계조차 없음) — 매우 제한적
Node failure Crash-stop crash로만 실패, 죽으면 영영 안 돌아옴
Crash-recovery crash 후 재시작 가능 — 안정 저장소는 보존, 메모리는 유실. 가장 유용한 현실적 모델
Byzantine (arbitrary) 무엇이든 할 수 있음 — 속임수 포함
  • 실무 표준 조합: partially synchronous + crash-recovery.
  • Safety: "나쁜 일이 일어나지 않는다" — 위반 시점을 특정할 수 있고 되돌릴 수 없다(예: fencing token의 uniqueness·단조 증가). crash·네트워크 유실을 포함한 모든 상황에서 항상 성립해야 한다.
  • Liveness: "좋은 일이 결국(eventually) 일어난다" (예: 요청은 결국 응답을 받는다). 조건부 보장이 허용된다(과반 노드 생존 + 네트워크가 결국 회복하면).
  • 이론 vs 실전: system model은 추상화다 — 현실에선 "안정 저장소"조차 깨질 수 있다(디스크 펌웨어 버그). 그래도 추상 모델에서의 정확성 증명은 무의미하지 않다 — 오히려 그 덕에 알고리즘을 여러 구현에서 신뢰할 수 있다. 증명과 현실 테스트가 둘 다 필요하다.

쉬운 예시 / 비유

Partial failure — 오케스트라의 한 명

독주자(단일 머신)는 틀리면 연주 전체를 멈추고 다시 시작하면 된다. 백 명의 오케스트라(분산 시스템)는 한 명이 악보를 놓쳐도 나머지는 연주를 계속한다 — 문제는 "누가 틀렸는지, 틀리긴 했는지"를 연주 중에 알기 어렵다는 것.

응답 없는 요청 — 답장 없는 편지

편지를 보냈는데 답이 없다. 편지가 유실됐나, 받고 무시했나, 답장이 유실됐나, 상대가 이사 갔나 — 답장이 오기 전까지는 구별할 방법이 없다. 기다리다 포기(timeout)할 수는 있지만, 상대가 읽었는지는 여전히 모른다.

Timeout 딜레마 — 회의에 늦는 동료

10분을 기다려도 안 오면 "결석" 처리하고 회의를 시작했는데 11분에 도착하면? 그 사람 몫의 일을 이미 남에게 배정했다면 같은 일을 두 명이 하게 된다. 그렇다고 1시간을 기다리면 모두의 시간 낭비다.

Queueing — 출근길 정체

도로(네트워크) 자체는 그대로인데 차가 몰리면(bursty traffic) 램프·교차로마다 줄이 생겨 소요 시간이 들쭉날쭉해진다. 전용 차선 예약(circuit switching)은 정시성을 주지만 빈 시간엔 아무도 못 쓴다 — 정시성 vs 도로 활용률의 trade-off.

두 시계 — 벽시계와 스톱워치

라면 끓이는 시간은 스톱워치(monotonic)로 재야 한다. 벽시계(time-of-day)는 재는 도중 누가 시간을 맞춰버리면(NTP 점프) "라면이 -30초 만에 익었다"는 결과가 나올 수 있다.

LWW — 심판마다 다른 초시계

각 지역 심판이 자기 손목시계로 기록을 찍는 대회. 시계가 느린 심판의 선수는 실제로 나중에 완주했어도 기록상으로는 먼저 들어온 선수를 이겨버린다 — 아무도 눈치채지 못한 채(silent data loss).

Process pause — 발표 중 졸도

발표자가 "이제 결론을…" 하다가 1분간 의식을 잃었다 깨어나면 본인은 문장을 잇지만, 청중은 이미 다른 발표자를 무대에 올렸다. 본인만 1분이 지난 것을 모른다 — GC pause 후의 leader가 정확히 이 상태다.

Quorum — 항해 중의 다수결

선장이 "나는 멀쩡하다"고 우겨도, 선원 과반이 "선장이 쓰러졌었다"고 판정하면 새 선장을 뽑는다. 개인의 자기 인식보다 다수의 합의가 공식 기록이 된다.

Fencing token — 은행 창구의 번호표

창구(storage)는 지금까지 처리한 번호보다 작은 번호표를 들고 오는 손님을 거절한다. 화장실 다녀오느라(GC pause) 옛 번호표를 들고 온 손님은 자신이 유효하다고 믿어도 창구에서 걸러진다 — 손님의 양심이 아니라 창구의 검사가 안전을 만든다.

Byzantine fault — 거짓말하는 장군

포위 공격의 성패는 장군들의 동시 공격에 달렸는데, 일부 장군이 배신해 서로 다른 전갈을 보낸다. 연락 두절 장군(crash)보다 거짓말하는 장군이 훨씬 어렵다. 데이터센터는 같은 편 장군만 있다고 가정하고(non-Byzantine), 블록체인은 배신자를 전제한다.

Safety vs liveness — 신호등의 두 약속

"교차 방향이 동시에 초록이 되지 않는다"는 safety(한 번이라도 깨지면 사고, 되돌릴 수 없음), "빨간불은 언젠가 초록이 된다"는 liveness(당장은 아니어도 결국). 안전은 항상, 진행은 조건부로.

용어 정의
Partial failure 시스템 일부만 망가지고 나머지는 동작 — 분산 시스템의 정의적 특징, 비결정적
HPC vs Cloud 접근 부분 실패를 전체 실패로 승격(checkpoint 재시작) vs fault를 전제로 소프트웨어에서 fault tolerance 구축

Network partition (netsplit) 네트워크 일부가 다른 부분과 단절되는 장애
Timeout 일정 시간 응답 없으면 실패 간주 — 원인은 구별 못 하고 그저 포기

Phi Accrual failure detector 응답 시간 분포로 timeout을 자동 조정 (Akka, Cassandra)
Circuit-switched / Packet-switched 대역폭 예약으로 bounded delay vs bursty traffic에 최적화된 큐잉 (활용률 극대화)

Clock drift 수정 발진기가 빠르거나 느리게 가는 현상
Leap second 59초/61초짜리 1분 — 실제 장애 원인, smearing으로 완화
LWW (last write wins) timestamp 최대 쓰기가 승리 — 시계 skew 시 조용한 데이터 유실·causality 위반
Logical clock 물리 시간 대신 단조 증가 카운터로 상대적 순서를 매김

Process pause GC, VM suspend, CPU steal, swap, SIGSTOP 등으로 임의 시간 정지
Lease 시한부 lock — pause 중 만료를 본인만 모를 수 있음
Hard real-time system deadline 보장 시스템(RTOS) — 서버 데이터 시스템에는 비경제적
Quorum 과반수 투표로 결정 — 개별 노드의 자기 판단보다 우선하는 "진실"
Fencing token lock 발급마다 단조 증가하는 토큰 — storage가 검사해 낡은 보유자의 쓰기 차단
Byzantine fault 노드가 임의로 오작동하거나 거짓말을 함 (비잔틴 장군 문제)
Byzantine fault tolerance 일부 노드의 배신에도 올바르게 동작 — 항공우주·블록체인용, 대부분의 데이터 시스템은 non-Byzantine 가정
Synchronous / Partially synchronous / Asynchronous 지연·pause·시계 오차의 bounded 여부에 대한 타이밍 모델
Crash-stop / Crash-recovery / Byzantine 노드 실패 모델 — 실무 표준은 partially synchronous + crash-recovery
Safety "나쁜 일이 일어나지 않는다" — 위반은 되돌릴 수 없고, 항상 성립해야 함
Liveness "좋은 일이 결국 일어난다" — eventually, 조건부 보장 허용

한 줄 요약

분산 시스템에서는 네트워크(unbounded delay), 시계(drift·점프), 프로세스(pause)를 모두 믿을 수 없고, 응답이 없으면 원인조차 구별할 수 없다 — 그래서 진실은 개별 노드가 아니라 quorum이 정의하고, 안전은 클라이언트의 양심이 아니라 fencing token 검사가 만들며, 알고리즘은 system model(partially synchronous + crash-recovery)의 가정 위에서 safety는 항상, liveness는 조건부로 보장하도록 설계한다.
