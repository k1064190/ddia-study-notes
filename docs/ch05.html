Ch5. Replication

복제의 진짜 난제는 정적 데이터가 아니라 변경(change)의 전파 — single-leader, multi-leader, leaderless라는 세 가지 접근과 그 트레이드오프

챕터 개요

Replication(복제)은 같은 데이터의 복사본을 네트워크로 연결된 여러 머신에 보관하는 것이다. 이유는 세 가지.

  • 지연(latency) 감소: 데이터를 사용자와 지리적으로 가까운 곳에 둔다.
  • 가용성(availability) 향상: 일부 노드가 죽어도 시스템이 계속 동작한다.
  • 읽기 처리량(read throughput) 증가: 읽기 요청을 여러 머신에 분산한다.

데이터가 절대 변하지 않는다면 복제는 쉽다 — 한 번 복사하면 끝. 진짜 어려운 것은 데이터의 변경(change)을 복제본에 반영하는 일이다. 이 챕터는 변경을 다루는 세 가지 대표 알고리즘 — single-leader, multi-leader, leaderless replication — 과 그 트레이드오프(sync/async, failover, replication lag, 충돌 해결 등)를 다룬다.

핵심 개념 정리

Single-leader (leader-based) replication

가장 흔한 방식 (PostgreSQL, MySQL, MongoDB, Kafka 등).

  • 복제본 중 하나를 leader(primary/master)로 지정한다. 쓰기는 반드시 leader로만 간다.
  • leader는 로컬에 쓰고, 변경 사항을 replication log(change stream)로 follower(replica/secondary)들에게 보낸다. follower는 같은 순서로 적용한다.
  • 읽기는 leader든 follower든 어디서나 가능 → 읽기 확장에 유리.

Synchronous vs Asynchronous replication

방식 장점 단점
Synchronous follower에 최신 복사본이 있음을 보장 — leader가 죽어도 데이터 손실 없음 follower가 응답하지 않으면 쓰기가 멈춘다 → 모든 follower를 동기로 두는 것은 비현실적
Asynchronous follower가 느리거나 죽어도 leader는 계속 쓸 수 있다 leader가 죽으면 아직 복제되지 않은 쓰기가 유실될 수 있다 (durability 약화)
Semi-synchronous follower 하나만 동기, 나머지는 비동기 — 최소 두 노드(leader + 동기 follower)에 최신 데이터 보장 동기 follower가 느려지면 다른 follower를 동기로 승격해야 함

새 follower 세팅 (Setting up new followers)

단순 파일 복사는 안 된다 — 복사 도중에도 데이터가 계속 변하기 때문. 표준 절차:

  1. leader의 일관된 스냅샷(consistent snapshot)을 뜬다 (가능하면 leader를 잠그지 않고).
  2. 스냅샷을 새 follower에 복사한다.
  3. follower가 스냅샷 시점 이후의 변경을 leader에게 요청한다 — 스냅샷이 replication log의 정확한 위치(PostgreSQL의 log sequence number, MySQL의 binlog coordinates)와 연결되어 있어야 한다.
  4. 밀린 변경을 다 따라잡으면(caught up) 정상 follower로 동작한다.

노드 장애 처리 (Handling node outages)

  • Follower 장애 — catch-up recovery: follower는 받은 log를 디스크에 보관하므로, 재시작 후 장애 직전 마지막으로 처리한 트랜잭션 위치부터 leader에게 이후 변경을 요청해 따라잡으면 된다.
  • Leader 장애 — failover: 훨씬 까다롭다. ① leader 사망 판정(보통 timeout 기반) → ② 새 leader 선출(보통 가장 최신 데이터를 가진 replica) → ③ 시스템 재구성(클라이언트 쓰기 라우팅 변경, 옛 leader 복귀 시 follower로 강등).
심층 설명 — failover의 함정: 자동 failover는 단순해 보이지만 곳곳이 지뢰밭이다. (1) 쓰기 유실 — 비동기 복제에서 새 leader가 옛 leader의 최신 쓰기를 아직 못 받았을 수 있고, 보통 그 쓰기를 조용히 버린다(durability 기대 위반). (2) 외부 시스템과의 불일치 — GitHub 사고에서는 뒤처진 follower가 승격되며 MySQL autoincrement primary key를 재사용했고, Redis에 이미 그 key로 저장된 데이터와 충돌해 다른 사용자의 개인 데이터가 노출됐다. (3) split brain — 두 노드가 모두 자기가 leader라고 믿고 쓰기를 받으면 데이터가 유실·훼손된다; 한쪽을 강제 종료하는 장치가 필요한데, 잘못 설계하면 둘 다 죽는다. (4) timeout 튜닝 딜레마 — 너무 길면 복구가 느리고, 너무 짧으면 부하 스파이크·네트워크 글리치에 불필요한 failover가 일어나 상황을 악화시킨다. 그래서 일부 운영팀은 자동 failover 대신 수동 failover를 선호한다.

Replication log 구현 방식

방식 아이디어 문제/특징
Statement-based SQL 문장 자체를 follower에 전달 비결정적(nondeterministic) 함수(NOW(), RAND()), autoincrement 의존, side effect 있는 트리거가 replica마다 다른 결과를 낼 수 있음 → 요즘은 잘 안 씀
WAL shipping storage engine의 write-ahead log 바이트를 그대로 전달 (PostgreSQL, Oracle) log가 디스크 저장 포맷과 밀결합 → leader/follower가 다른 버전을 못 쓰는 경우가 많아 무중단 버전 업그레이드가 어려움
Logical (row-based) log row 단위 변경(삽입 row의 전체 값, 삭제/갱신 row의 식별자와 변경 값)을 기록 (MySQL binlog row 모드) storage engine 내부와 분리(decoupled) → 버전 호환·다른 storage engine 간 복제 가능. 외부 시스템 전송에도 유용 (change data capture)
Trigger-based 애플리케이션 레벨 트리거로 변경을 별도 테이블에 기록, 외부 프로세스가 복제 (Databus, Bucardo) 유연하지만 오버헤드가 크고 버그에 취약

Replication lag과 세 가지 읽기 일관성

비동기 follower에서 읽으면 stale(오래된) 데이터를 볼 수 있다. 기다리면 결국 일치한다 — eventual consistency. 하지만 lag이 커지면 실제 문제가 된다.

보장 막는 현상 구현 방법
Read-after-write (read-your-writes) 자기가 쓴 글이 새로고침 후 안 보이는 것 수정 가능한 데이터(자기 프로필)는 leader에서 읽기; 최근 쓰기 후 일정 시간 leader에서 읽기; 최근 쓰기의 timestamp를 기억해 그 시점까지 따라잡은 replica에서만 읽기. cross-device(여러 기기)에서는 timestamp 공유·datacenter 라우팅 문제로 더 어렵다
Monotonic reads 방금 봤던 데이터가 다음 읽기에서 사라지는 등 시간이 거꾸로 가는 현상 각 사용자를 항상 같은 replica에서 읽게 한다 (예: user ID의 hash로 replica 선택)
Consistent prefix reads 질문보다 답변이 먼저 보이는 등 인과 관계(causal order)가 뒤집히는 현상 — partitioned 시스템에서 파티션마다 독립적으로 쓰기가 적용되어 발생 인과적으로 연결된 쓰기를 같은 파티션에 쓰거나, 인과성을 추적하는 알고리즘 사용

→ 애플리케이션이 이런 보장을 직접 구현하는 것은 복잡하다. 근본 해법은 transaction이지만, 분산 환경에서는 비용 문제로 포기하는 시스템이 많다 (7장·9장에서 계속).

Multi-leader replication

leader를 여러 개 두고, 각 leader는 다른 leader의 follower도 된다. 단일 datacenter 안에서는 복잡성 대비 이득이 적어 잘 안 쓰고, 다음 상황에서 의미가 있다.

  • Multi-datacenter 운영: datacenter마다 leader 하나 — 쓰기가 로컬에서 처리되어 지연이 낮고, datacenter 장애·네트워크 문제에 강하다.
  • 오프라인 동작 클라이언트: 캘린더 앱처럼 각 기기가 로컬 DB(= 사실상의 leader)를 갖고 오프라인에서 쓰고, 온라인 시 동기화.
  • 협업 편집(collaborative editing): Google Docs 등 — 각 사용자의 로컬 변경이 비동기로 다른 사용자에게 복제.

Topology: circular(원형)·star(별형)는 한 노드 장애 시 전파가 끊길 수 있음; all-to-all은 가장 일반적이지만 네트워크 속도 차이로 메시지가 추월해 인과 순서 문제가 생길 수 있다 (version vector 등 필요).

쓰기 충돌(write conflict) — multi-leader의 최대 난제. 두 leader가 같은 데이터를 동시에 수정하면 충돌이 비동기적으로 나중에 감지된다.

  • 충돌 회피(avoidance): 특정 레코드의 쓰기를 항상 같은 leader로 라우팅 (예: 사용자별 home datacenter). 가장 흔한 실전 전략.
  • Convergence(수렴): 모든 replica가 결국 같은 값에 도달해야 한다 — Last write wins (LWW)(가장 큰 ID/timestamp가 이김; 단순하지만 데이터 유실이 조용히 일어난다), 값들의 merge(예: "B/C"), 충돌을 명시적으로 기록 후 해결.
  • Custom conflict resolution: 애플리케이션 코드로 — on write(충돌 감지 즉시 핸들러 실행, 예: Bucardo) 또는 on read(충돌 버전을 모두 저장했다가 읽을 때 해결, 예: CouchDB).
  • 자동 해결 연구: CRDT(conflict-free replicated data types), mergeable persistent data structures, operational transformation(Google Docs).

Leaderless replication (Dynamo-style)

leader가 없고 클라이언트가 여러 replica에 직접(또는 coordinator를 통해) 쓴다. Amazon Dynamo에서 유래 — Cassandra, Riak, Voldemort 등.

  • Quorum 읽기/쓰기: replica가 n개일 때, 쓰기는 w개의 확인, 읽기는 r개의 응답을 받는다. w + r > n이면 읽은 노드 중 적어도 하나는 최신 값을 갖는다 (예: n=3, w=2, r=2). w·r 조절로 트레이드오프 조정 가능 (읽기 많은 워크로드는 w=n, r=1 등).
  • Read repair: 읽기 시 여러 replica의 응답을 비교해, 뒤처진 replica에 최신 값을 써준다.
  • Anti-entropy process: 백그라운드 프로세스가 replica 간 차이를 찾아 복사한다. 이게 없으면 잘 안 읽히는 값은 오래도록 stale.
  • Sloppy quorum & hinted handoff: 네트워크 단절로 지정된 "home" 노드들에 도달하지 못하면 다른 노드에 임시로 쓰기를 받아둔다(sloppy quorum). 복구 후 임시 노드가 home 노드로 데이터를 넘긴다(hinted handoff). 쓰기 가용성은 높아지지만, w + r > n이어도 최신 값 읽기를 보장하지 못하게 된다.
  • Multi-datacenter: Cassandra는 n을 전체 datacenter에 걸쳐 두되 quorum 확인은 로컬 datacenter에서만 기다리는 설정 가능. Riak은 datacenter 간을 별도의 비동기 복제로 처리.
심층 설명 — quorum이 만능이 아닌 이유: w + r > n은 "읽기 집합과 쓰기 집합이 겹친다"는 집합 논리일 뿐, 절대적 보장이 아니다. edge case들: (1) sloppy quorum에서는 쓰기 w개가 읽기 r개와 다른 노드 집합에 갈 수 있어 겹침 자체가 깨진다. (2) 동시 쓰기가 일어나면 어느 쪽이 이겼는지 모호하고, LWW로 자르면 clock skew 때문에 쓰기가 유실될 수 있다. (3) 쓰기와 읽기가 동시에 일어나면 읽기가 신·구 값 중 무엇을 볼지 미정이다. (4) 쓰기가 일부 replica에서만 성공하고 실패로 보고돼도 성공한 replica에서는 롤백되지 않는다. (5) 최신 값을 가진 노드가 죽고 옛 값으로 복구되면 w 조건이 깨질 수 있다. 결론: quorum은 read-your-writes나 monotonic reads도 보장하지 않으며, staleness를 모니터링하는 것이 실무적으로 중요하다.

동시 쓰기 감지 (Detecting concurrent writes)

  • 핵심 질문: 두 작업 A, B 중 하나가 먼저인가(happens-before), 동시(concurrent)인가? B가 A를 알고 있거나, 의존하거나, 기반으로 하면 A → B. 서로를 모르면 concurrent — 물리적 시각이 겹치는지가 아니라 서로의 존재를 몰랐는지가 기준.
  • Version number 알고리즘 (단일 replica): 서버가 key마다 version number를 유지하고 쓰기마다 증가. 클라이언트는 쓰기 전에 반드시 읽고, 쓸 때 직전에 읽은 version number를 함께 보낸다. 서버는 그 version 이하의 값들은 덮어쓰고, 그보다 높은 version의 값은 concurrent이므로 sibling으로 함께 보관한다.
  • Sibling 병합 (장바구니 예시): Dynamo 장바구니 — sibling들을 클라이언트가 union으로 병합. 단, 단순 union이면 지운 상품이 부활한다 → 삭제를 tombstone(삭제 표시)으로 기록해야 한다. 이런 병합은 오류가 나기 쉬워 CRDT가 자동 병합 자료구조를 제공한다.
  • Version vector: replica가 여러 개면 replica별 version number를 모아 관리 (Riak의 causal context). replica 간에도 덮어쓰기 가능 여부와 동시성(sibling)을 판별하게 해준다.
심층 설명 — version vector와 동시 쓰기 감지: 여기서 "concurrent"는 시계의 문제가 아니라 정보의 문제다. 클라이언트가 쓰기에 "직전에 읽은 version"을 실어 보내는 순간, 그 쓰기는 "나는 여기까지 알고 썼다"는 선언이 된다. 서버는 이 선언 덕분에 그 version 이하의 값은 안전하게 덮어쓰고(클라이언트가 이미 반영했으므로), 선언보다 높은 version의 값은 서로 모른 채 쓰인 것이므로 sibling으로 보존한다. replica가 여러 개가 되면 version number 하나로는 부족하다 — replica A의 3번째 쓰기와 replica B의 5번째 쓰기는 비교 자체가 안 되기 때문이다. 그래서 replica별 counter의 벡터를 통째로 들고 다니는 것이 version vector다: 한 벡터가 다른 벡터의 모든 성분보다 크거나 같으면 happens-before(덮어쓰기 가능), 서로 엇갈리면 concurrent(sibling 보존). 흔한 오해는 "timestamp가 더 크면 나중 쓰기"라는 것 — 그게 바로 LWW의 함정이고, version vector는 시계 없이 인과성만으로 판별한다.

쉬운 예시 / 비유

Single-leader — 칠판과 필기 노트

선생님(leader) 혼자 칠판에 쓰고, 학생들(follower)은 그대로 받아 적는다. 조회(읽기)는 아무 노트나 보면 되지만, 내용 변경(쓰기)은 선생님만 한다. 필기가 느린 학생의 노트는 잠깐 뒤처진다(replication lag).

Sync vs Async — 등기우편 vs 일반우편

동기 복제는 수신 확인을 받을 때까지 기다리는 등기우편 — 확실하지만 상대가 부재중이면 하염없이 기다린다. 비동기 복제는 일반우편 — 빠르지만 도착 보장이 없다.

Split brain — 선장이 둘인 배

선장이 쓰러진 줄 알고 부선장이 지휘를 시작했는데, 사실 선장은 잠깐 정신을 잃었을 뿐. 둘 다 서로 다른 명령(쓰기)을 내리면 배는 산으로 간다.

Read-your-writes — 내가 쓴 방명록

글을 남기고 새로고침했는데 내 글이 안 보이면 "지워졌나?" 하고 다시 쓴다. 남의 글은 늦게 보여도 되지만, 내 글은 나에게 즉시 보여야 한다.

Monotonic reads — 되감기 금지

영화 장면이 갑자기 5분 전으로 돌아가면 이상하다. 최신 상태를 봤다면 그보다 과거 상태를 다시 보여주면 안 된다.

Consistent prefix reads — 질문보다 먼저 들리는 답변

답변("약 10초요")이 질문("얼마나 앞서 보이나요?")보다 먼저 들리면 인과가 뒤집혀 대화가 이상해진다.

Quorum — 도장 3개 중 2개

서류(쓰기)에 위원 3명 중 2명의 도장을 받고, 조회(읽기)할 때도 2명에게 물어보면, 두 집합이 반드시 겹치므로 적어도 한 명은 최신 서류를 안다 (w + r > n).

Sloppy quorum & hinted handoff — 옆집에 맡긴 택배

집(home 노드)에 아무도 없으면 옆집(임시 노드)에 맡긴다. 집주인이 돌아오면 옆집이 전달한다(hinted handoff). 단, 그 사이에 집에 가서 물어보면 택배가 없다고 답한다 — 최신 값 읽기 보장이 깨지는 이유.

Version vector — 각자 세는 회의록 번호

회의록을 여러 서기(replica)가 따로 쓰는데, 서기별 개정 번호를 전부 들고 다니면 "이 판본이 저 판본을 이미 반영했는지(happens-before), 서로 모르는 판본인지(concurrent)"를 판별할 수 있다.

핵심 용어 정리 (Glossary)

용어 정의
Replication 같은 데이터의 복사본을 여러 노드에 보관 (지연↓, 가용성↑, 읽기 처리량↑)
Leader / Follower 쓰기를 받는 노드 / leader의 변경을 받아 적용하는 노드
Replication log leader가 follower에게 보내는 변경 스트림
Synchronous / Asynchronous follower 확인을 기다림(안전, 느림) / 안 기다림(빠름, 유실 가능)
Semi-synchronous follower 하나만 동기, 나머지는 비동기로 두는 절충
Catch-up recovery follower가 장애 후 마지막 처리 위치부터 변경을 따라잡는 복구
Failover leader 장애 시 follower 하나를 새 leader로 승격하는 과정
Split brain 두 노드가 동시에 자신을 leader라 믿는 위험 상태
Statement-based replication SQL 문장을 그대로 전달 (비결정성 문제)
WAL shipping write-ahead log 바이트를 전달 (저장 포맷과 밀결합)
Logical (row-based) log row 단위 변경을 storage engine과 분리해 기록
Change data capture DB 변경 스트림을 외부 시스템으로 내보내는 기법
Replication lag leader와 follower 사이의 반영 지연
Eventual consistency 쓰기를 멈추면 모든 replica가 결국 같은 값에 수렴
Read-after-write consistency 사용자가 자기가 쓴 데이터는 항상 볼 수 있음
Monotonic reads 한 번 본 상태보다 과거 상태를 다시 보지 않음
Consistent prefix reads 인과 순서가 있는 쓰기를 그 순서대로 봄
Multi-leader replication 쓰기를 받는 leader가 여러 개 (multi-datacenter, 오프라인, 협업 편집)
Write conflict 두 leader가 같은 데이터를 동시에 수정해 생기는 충돌
Last write wins (LWW) timestamp가 가장 큰 쓰기만 남기고 나머지를 조용히 버림
CRDT 자동으로 합리적 병합이 되는 conflict-free replicated data type
Leaderless replication leader 없이 클라이언트가 여러 replica에 직접 읽고 씀 (Dynamo-style)
Quorum (w + r > n) 쓰기 w개·읽기 r개 확인으로 읽기 집합에 최신 값 포함을 노림
Read repair 읽기 시 stale replica를 발견해 최신 값으로 고쳐 씀
Anti-entropy 백그라운드로 replica 간 차이를 찾아 동기화하는 프로세스
Sloppy quorum home 노드 대신 도달 가능한 다른 노드에 임시 쓰기를 허용
Hinted handoff 임시 노드가 받아둔 쓰기를 네트워크 복구 후 home 노드로 전달
Happens-before / Concurrent 한 작업이 다른 작업을 알고 있음 / 서로의 존재를 모름
Version vector replica별 version number 모음 — 덮어쓰기 가능 여부와 동시성 판별
Sibling 동시 쓰기로 생겨 함께 보관되는 충돌 값들
Tombstone 삭제를 기록하는 표시 (sibling 병합 시 삭제 부활 방지)

한 줄 요약

복제의 진짜 난제는 변경의 전파다 — single-leader는 단순하지만 failover와 replication lag(read-your-writes·monotonic reads·consistent prefix)이 문제고, multi-leader와 leaderless(quorum, w + r > n)는 가용성을 얻는 대신 동시 쓰기 충돌을 감지(version vector)하고 해결(merge, LWW, CRDT)해야 한다.
