Ch11. Stream Processing

batch의 인위적 경계(고정·유한 입력)를 걷어내고 끝없이 도착하는 불변 event를 다루는 패러다임 — message broker(전통적 → Kafka류 log 기반), CDC와 event sourcing, 그리고 시간·join·fault tolerance의 세계

챕터 개요

Chapter 10의 batch processing은 강력하지만 한 가지 가정을 깐다 — 입력이 유한(bounded)하고 고정되어 있다는 것이다. batch job은 "어제 자정까지의 모든 데이터"처럼 정해진 입력을 읽어 정해진 출력을 낸다. 하지만 현실의 데이터는 끝없이 도착한다 — 사용자는 밤낮으로 페이지를 열고 센서는 계속 값을 뿜는다. 하루 단위 batch는 결과를 보기까지 최대 하루가 걸리니, 처리 간격을 잘게 쪼개다 보면 결국 고정된 시간 조각이라는 개념 자체를 버리고, 이벤트가 발생할 때마다 처리하는 지점에 다다른다 — 이것이 stream processing이다.

이 챕터는 세 부분이다: Transmitting Event Streams(event를 producer에서 consumer로 나르기 — message broker의 두 패턴, log 기반 broker), Databases and Streams(CDC와 event sourcing으로 시스템을 동기화하고 불변 log를 진실의 원천으로 삼기), Processing Streams(event time vs processing time, windowing, stream join, fault tolerance).

핵심 통찰 — batch와 stream은 대립이 아니라 연속체다. batch는 bounded stream이고 stream은 unbounded batch다. 불변 event log를 진실의 원천으로 두고 거기서 다양한 read view를 파생시킨다는 발상이 이 챕터 전체를 관통한다.

핵심 개념 정리

왜 stream인가 — batch의 인위적 경계

batch의 입력은 bounded하다 — 유한하고 언젠가 끝나므로 job이 "언제 다 읽었는지" 안다. 하지만 이 유한성은 데이터를 하루 단위로 인위적으로 잘라낸 결과일 뿐이다. 실제 데이터는 점진적으로(incrementally) 도착하는 unbounded stream이며, daily batch의 문제는 하루의 지연이다. 처리 주기를 시간→분→초로 쪼개다 보면 고정 chunk를 버리고 이벤트마다 처리하게 된다.

event: stream의 기본 단위 — 어떤 시점에 일어난 일에 대한, 작고 자기완결적(self-contained)이며 불변인 record. 보통 timestamp를 포함하고 텍스트·JSON·binary로 인코딩된다. event를 만드는 쪽이 producer(publisher), 처리하는 쪽이 consumer(subscriber)이며, 관련 event를 topic으로 묶는다.

Event stream의 전송 — messaging system

낮은 지연이 목표라면 consumer가 매번 polling하는 것은 비효율적이다 — 간격을 좁힐수록 "새 것 없음" 비용이 커진다. 그래서 새 event를 밀어주는(notify) messaging system이 필요하다. pub/sub 모델은 두 질문으로 특징지어진다: ① producer가 빠르면 어떻게 하나(버림 / 큐잉 / backpressure), ② 노드가 죽으면 event가 유실되나. UDP 멀티캐스트·브로커 없는 방식·webhook 같은 direct producer→consumer는 단순하지만 consumer의 일시적 장애에 취약하고 메시지 유실 위험이 크다.

Message broker: producer와 consumer 사이에 두는 서버 — 사실상 message에 최적화된 데이터베이스다. broker에 데이터가 쌓이므로 consumer는 비동기·간헐적이어도 되고, 죽었다 살아나도 놓친 메시지를 받는다. 두 가지 소비 패턴:

패턴 동작 용도
Load balancing
(shared subscription)		 각 메시지를 consumer 하나에게만 전달, 여럿이 나눠 처리 처리량 확장 — 메시지 처리가 비쌀 때 병렬화
Fan-out 각 메시지를 모든 consumer에게 전달 여러 독립 consumer가 같은 stream을 각자 소비

두 패턴은 조합 가능하다(consumer group 여러 개, 각 group 안에서 load balancing). consumer는 언제든 죽으므로 broker는 acknowledgment(처리 완료 통지)를 쓴다 — ack 없이 연결이 끊기면 redeliver한다. 문제: load balancing과 결합되면 메시지 순서가 뒤바뀔 수 있다 — consumer2가 m3를 처리하다 죽어 재전달되는 사이 consumer1이 m4, m5를 이미 처리했다면 순서가 섞인다.

심층 설명 — log 기반 broker vs 전통적 broker, 그리고 replay: 전통적 broker(AMQP/JMS)는 메시지를 소비·ack하면 삭제한다 — 과거를 다시 읽을 수 없고, 개별 메시지마다 ack를 추적한다. Kafka류 log 기반 broker는 정반대 발상이다 — 디스크의 append-only log를 그대로 broker로 쓴다. producer는 log 끝에 append하고, consumer는 log를 순차적으로 읽는다(끝에 닿으면 새 메시지를 대기). 확장을 위해 log를 partition으로 나누며, 한 partition 안에서는 순서가 보장되고 각 메시지에 단조 증가하는 offset이 붙는다. 소비 패턴은 자연히 매핑된다 — fan-out은 consumer들이 서로 독립적으로 log를 읽으면 되고(누구도 지우지 않으므로), load balancing은 partition 단위로 consumer group에 분배한다 (그래서 consumer 수 ≤ partition 수, 느린 메시지는 그 partition을 head-of-line blocking하지만 순서는 보존). broker는 개별 ack 대신 consumer가 어디까지 읽었는지 offset 하나만 기록한다(single-leader DB의 log sequence number와 같은 발상) — 죽으면 마지막 커밋 offset부터 재개하되 그 사이 메시지는 재처리될 수 있다. 가장 큰 강점은 replay다 — 소비해도 지우지 않으므로 offset을 과거로 되돌려 옛 메시지를 다시 소비할 수 있다(파생 시스템 재생성, 코드 수정 후 재처리, 실험). append-only라 디스크가 결국 차므로, 원형 버퍼로 오래된 segment를 지우거나(느린 consumer가 너무 뒤처지면 메시지 유실), key별 최신 값만 남기는 log compaction으로 log 안에 전체 스냅샷을 유지한다.

언제 무엇을 쓰나: 메시지 처리가 비싸고 메시지별 병렬화가 필요하며 순서가 덜 중요하면 전통적(AMQP/JMS) broker가, 처리가 빠르고 순서가 중요하며 replay가 필요하면 log 기반 broker가 낫다.

데이터베이스와 stream — CDC와 event sourcing

현실에는 OLTP DB, 검색 인덱스, 캐시, 데이터 웨어하우스 등 같은 데이터의 여러 사본이 공존하고 이들을 일치시켜야 한다. 애플리케이션이 각 시스템에 직접 이중 쓰기(dual write)하는 방식은 위험하다:

  • race condition: 두 클라이언트의 동시 쓰기가 DB에는 A→B, 검색 인덱스에는 B→A 순서로 도착해 두 시스템이 영구히 불일치할 수 있다.
  • partial failure: 한 쓰기는 성공, 다른 쓰기는 실패하면 어긋난다 — atomic commit(2PC)으로 막으려면 비싸다.

해법은 하나의 시스템을 진실의 원천(source of truth)인 leader로 정하고 나머지를 그 변경 로그를 따라가는 follower로 만드는 것이다. Change data capture(CDC)는 DB에 쓰인 모든 변경을 추출해 stream으로 만들어, 검색 인덱스·캐시·웨어하우스가 같은 순서로 적용하게 한다.

측면 내용
구현 방식 log 기반 CDC(replication log/WAL 파싱 — Debezium이 MySQL binlog·Postgres WAL을 읽음) vs trigger 기반 CDC(DB trigger로 변경 기록 — 이식성은 좋으나 오버헤드·취약)
일관성 본질적으로 asynchronous — follower가 leader보다 뒤처질 수 있으나 dual write의 race·partial failure가 없다(하나의 순서 있는 log)
초기 스냅샷 전체 변경 log를 영원히 못 두므로, 일관된 스냅샷을 먼저 적재하고 그 이후 log를 이어 붙임(스냅샷은 log의 알려진 offset에 대응)
log compaction key별 최신 값만 남기면 log 하나로 DB 스냅샷 재구성 — 새 follower를 스냅샷 없이 log 처음부터 생성 가능
심층 설명 — CDC와 event sourcing의 관계·차이: 둘 다 "변경을 순서 있는 log로 흘린다"는 발상을 공유하지만 추상화 수준이 다르다. CDC는 DB의 저수준 상태 변경을 추출한다 — 이 event는 mutable하다(같은 row가 나중에 UPDATE로 덮이므로), 그래서 로그를 영구 보관할 필요는 없고 파생 시스템 동기화가 목적이다. event sourcing은 애플리케이션이 스스로 상태 변경을 애플리케이션 수준의 불변 event("학생이 수업에서 취소함")로 append-only log에 기록한다 — 여기서는 log 자체가 진실이고, 현재 상태는 event를 재생(replay)해 계산한다(매번 전부 재생은 비싸니 스냅샷을 캐시). 중요한 구별: command vs event — 사용자의 요청은 처음엔 command(검증 대상, 실패 가능)지만, 검증을 통과하면 되돌릴 수 없는 사실인 event가 되어 log에 들어간다. event는 이미 확정된 사실이므로 consumer가 재생 시 실패해서는 안 된다(검증은 command 단계에서 끝냈어야 한다). 이 발상을 밀고 나가면 불변 event log가 진실의 원천이고 현재 상태는 파생 view라는 결론에 이른다 — 회계 장부처럼 잘못은 지우지 않고 정정 event를 덧붙이며, 하나의 log에서 질의 패턴별로 최적화된 여러 read view를 파생시킨다. 이것이 CQRS(Command Query Responsibility Segregation)다.

State, streams, immutability의 장단점: 장점 — 감사·디버깅에 유리(완전한 이력), 과거 상태 재구성, 새 view 소급 생성, 버그로 잘못 쓴 view를 log에서 재구성해 복구. 한계 — log가 무한히 자라 compaction이 필요하고, 불변성은 삭제 요구(GDPR "잊힐 권리")와 충돌한다(진짜 삭제는 log에서 실제로 지워야 하므로 — 암호화 키 파기 등 — 간단치 않다).

stream의 처리 — 용도

  • Complex event processing (CEP): SQL 같은 질의로 event 패턴을 기술하고, 맞으면 complex event를 발생 — DB와 뒤집힌 관계(질의가 오래 머물고 데이터가 흘러 지나간다). 사기 탐지 등.
  • Stream analytics: 개별 event보다 window 단위 집계·통계(rate·평균·백분위)에 관심 — 근사 알고리즘(HyperLogLog, Bloom filter)을 자주 쓴다.
  • Materialized view 유지: 캐시·검색 인덱스·웨어하우스를 최신으로(CDC·event sourcing과 이어짐).
  • Search on streams: 미리 등록한 질의에 흘러오는 문서를 매칭(미디어 모니터링, 부동산 알림).
  • actor framework와 구별: actor는 동시성·분산 실행의 메커니즘이지 데이터 파이프라인 도구가 아니다(전달·순서 보장이 다름).
심층 설명 — event time vs processing time과 windowing: stream 처리에서 시간은 미묘하다. event time은 event가 실제 일어난 시각(device의 clock), processing time은 처리기가 그 event를 본 시각(server의 clock)이며, 둘은 큐잉·네트워크 지연·재처리로 크게 벌어진다. processing time으로 window를 나누면 처리 지연(재시작 후 밀린 event 몰림)이 실제 event rate 변화처럼 거짓 패턴으로 나타나므로, 정확하려면 event time으로 나눠야 한다. 그런데 event time 기반은 straggler 문제를 낳는다 — "이 window의 event가 이제 다 왔다"고 확신할 수 없다(늦게 오는 event가 있을 수 있다). 해법은 window를 언제 닫을지 timeout을 두고, 그 후 도착한 straggler는 무시(drop)하거나 이미 낸 결과를 정정(correction)하는 것이다. 게다가 device clock을 못 믿을 때(mobile)는 event에 세 timestamp — ① device의 발생 시각, ② device가 보낸 시각, ③ 서버가 받은 시각 — 를 실어, ②·③ 차이로 device clock offset을 추정해 ①을 보정한다. window 종류: tumbling(고정 길이·비겹침), hopping(고정 길이·겹침 허용), sliding(고정 경계 없이 이동), session(비활동 gap으로 구분, 가변 길이).

세 join 모두 한쪽의 상태가 시간에 따라 변한다는 문제가 있다 — 판매 시점의 세율로 세금을 계산해야 하는데 재처리 시점엔 세율이 바뀌었을 수 있다. 데이터 웨어하우스는 이를 slowly changing dimension(SCD)로 다뤄, 각 버전에 고유 식별자를 붙여 "그때 그 시점"의 값으로 결정론적 join을 한다(재처리 시 재현성 확보).

Fault tolerance — exactly-once

batch는 실패한 task를 통째로 재실행하면 되지만(입력 불변, 출력 버리고 다시), stream은 입력이 무한이라 "끝까지 처리 후 출력"이 불가능하다. 목표는 exactly-once(더 정확히 effectively-once) — 장애가 나도 결과가 한 번 처리한 것과 같아야 한다.

  • microbatching (Spark Streaming): stream을 아주 작은 batch(예: 1초)로 쪼개 batch의 fault tolerance를 재사용 — 실패 시 재실행.
  • checkpointing (Apache Flink): 주기적으로 연산자 상태의 일관된 checkpoint를 내구성 저장소에 기록, 장애 시 마지막 checkpoint부터 재개.
  • atomic commit: microbatch·checkpoint만으로는 부족하다 — 재시작하면 checkpoint 이후 출력이 외부로 다시 나갈 수 있다. exactly-once가 되려면 출력·상태 갱신이 원자적으로 커밋돼야 한다.
  • idempotence(멱등성): 같은 연산을 여러 번 해도 한 번과 같은 효과 — offset을 담아 "이미 반영됨"을 표시해 재처리 중복을 건너뛴다. 값싼 exactly-once 수단.
  • 상태 재구축: window·join·집계 상태가 장애 후에도 살아남아야 한다 — 원격 복제 또는 로컬 상태 + 변경 log 내구성 기록(Flink는 연산자 상태를 스냅샷).

쉬운 예시 / 비유

Batch vs stream — 사진 앨범 vs 라이브 방송

batch는 "어제까지 찍은 사진을 모아 한 권으로 인화" — 언제 끝났는지 알기에 목차가 깔끔하다. stream은 끊기지 않는 생방송 — "다 모은 다음"이라는 사치가 없다. batch는 그저 끝이 정해진 stream일 뿐이다.

Event — 영수증 한 장

event는 "언제 무슨 일이 있었는지" 적은 작은 영수증 — 이미 벌어진 일이라 고쳐 쓰지 않고, 정정이 필요하면 새 영수증(취소전표)을 덧붙인다.

Direct messaging의 위험 — 부재중 택배

초인종만 누르고 문 앞에 두고 가는 택배 — 집에 없으면(consumer offline) 잃어버린다. broker는 경비실(택배함) — 내가 없어도 맡아뒀다가 돌아오면 준다.

Load balancing vs fan-out — 콜센터 vs 사내 공지

load balancing은 콜센터 — 전화 하나를 빈 상담원 한 명에게만 돌린다. fan-out은 전 직원 공지 메일 — 모두에게 같은 내용이 간다.

Ack·redelivery의 순서 뒤섞임 — 회전초밥

손님(consumer)이 접시(m3)를 집다 놓쳐 다시 돌려보내는(redeliver) 사이 옆 손님이 이미 뒤 접시(m4, m5)를 집었다 — 원래 순서가 깨진다.

Log 기반 broker — 신문 합본 vs 우유 배달

전통적 broker는 우유 배달 — 마시면(ack) 병이 사라져 어제 것을 다시 못 마신다. Kafka는 신문 축쇄판 — 모두가 같은 지면을 각자의 속도로 읽고 지난 호를 언제든 다시 펼쳐 본다(replay). 어디까지 읽었는지는 책갈피(offset) 하나로 기억한다.

CDC — 대표 장부와 필사본

본사 원장(source-of-truth DB)에 기입이 생길 때마다 그 순서 그대로 지점들(검색 인덱스·캐시)의 필사본에 옮겨 적는다. 각자 따로 적어(dual write) 순서가 엇갈리는 사고를 원천 차단한다.

Event sourcing — 장바구니 로그 vs 현재 장바구니

"사과 담음 → 우유 담음 → 사과 뺌"이라는 행위의 기록만 남기고, "지금 장바구니 = 우유 1개"는 그 기록을 되짚어 계산한다. 기록이 진실이고 현재 상태는 파생물이다.

Event time vs processing time — 뒤늦게 도착한 편지

친구가 1일에 쓴 엽서(event time)가 파업으로 10일에 도착(processing time)했다. "이번 주에 받은 우편" 으로 세면 왜곡된다 — 엽서에 적힌 날짜로 분류해야 하지만 "1일자 엽서가 이제 다 왔나?"는 확신할 수 없다(straggler).

Windowing — 지하철 개찰구 집계

tumbling은 정각마다 끊는 시간대별 집계, hopping은 겹쳐 보는 이동 평균, session은 한 사람이 들어와 나갈 때까지를 한 묶음(gap이 길면 새 세션).

Stream-table join — 명찰 붙이기

출입 기록(stream)이 흘러올 때마다 사원 명부(table)를 뒤져 이름·부서를 덧붙인다(enrichment). 명부가 자주 바뀌므로 매번 원격 조회 대신 명부 변경(CDC)을 받아 로컬 사본을 최신으로 유지한다.

Exactly-once — 계좌 이체 재시도

송금 요청이 timeout돼 재시도했는데 두 번 빠져나가면 안 된다. 요청에 고유 번호를 달아 "이미 처리됨"으로 거르면(idempotence), 몇 번을 재시도해도 딱 한 번의 효과만 남는다.

한 줄 요약

Stream processing은 batch의 인위적 경계(고정·유한 입력)를 걷어내고 끝없이 도착하는 불변 event를 다루는 패러다임이다 — event를 나르는 message broker는 전통적 방식(소비 후 삭제)에서 log 기반(Kafka: append-only, offset, replay)으로 진화했고, DB 변경을 흘리는 CDC와 앱 event를 진실의 원천으로 삼는 event sourcing은 불변 log 하나에서 여러 read view를 파생(CQRS)한다는 통찰로 이어지며, 처리 단계에서는 event time vs processing time과 windowing 으로 시간을 다루고, stream join의 시간 의존성을 SCD로 통제하며, microbatching·checkpointing·idempotence로 exactly-once fault tolerance를 얻는다.
