Ch7. Transactions

온갖 fault와 동시성 문제를 abort 하나로 단순화하는 추상화 — ACID의 미끄러운 의미, weak isolation의 함정들, 그리고 serializability의 세 가지 구현

챕터 개요

Transaction(트랜잭션)은 여러 읽기·쓰기를 하나의 논리적 단위로 묶는 방법이다. 전체가 성공(commit)하거나 전체가 실패(abort, rollback)한다 — 부분 실패가 없으므로 애플리케이션은 "어중간하게 반쯤 적용된 상태"를 걱정할 필요가 없다.

데이터 시스템에서는 온갖 것이 잘못될 수 있다: 쓰기 도중 DB·애플리케이션 crash, 네트워크 단절, 여러 클라이언트의 동시 쓰기, 부분적으로만 갱신된 데이터 읽기, race condition 버그. 트랜잭션은 이런 문제들을 애플리케이션이 아니라 DB가 흡수해 프로그래밍 모델을 단순화하는, 수십 년간 다듬어진 추상화다. 다만 공짜가 아니다 — 이 챕터는 트랜잭션이 주는 보장(ACID), 그 보장이 실제로는 얼마나 약한지(weak isolation levels), 그리고 완전한 보장(serializability)을 구현하는 세 가지 방법을 다룬다.

핵심 개념 정리

ACID의 미끄러운 의미

오늘날 "ACID 호환"이라는 말은 기술 용어라기보다 마케팅 용어에 가깝다 — DB마다 구현이 제각각이고, 특히 isolation의 의미가 모호하다. ACID가 아닌 시스템을 BASE(Basically Available, Soft state, Eventual consistency)라 부르지만, 이는 "ACID가 아닌 모든 것"이라는 뜻일 뿐 더 모호하다.
글자 실제 의미 주의점
Atomicity fault(crash, 네트워크 단절 등) 시 트랜잭션의 쓰기를 전부 취소(abort)할 수 있음 — abortability, all-or-nothing 동시성과는 무관하다 (동시성은 Isolation의 몫). abort 덕분에 안전한 재시도가 가능해진다
Consistency 데이터에 대한 애플리케이션 수준의 불변식(invariant)(예: 회계 장부의 대차 일치)이 항상 참 불변식을 지키는 것은 애플리케이션의 책임 — DB는 저장할 뿐. "C는 사실 ACID에 속하지 않는다"; Joe Hellerstein에 따르면 acronym을 만들려고 끼워 넣어진 글자다
Isolation 동시 트랜잭션이 서로를 방해하지 않음 — 교과서적 이상은 serializability(순차 실행과 같은 결과) 성능 비용 때문에 serializable은 기본값인 경우가 드물고, 실무는 대부분 더 약한 격리 수준을 쓴다
Durability commit된 데이터는 crash에도 보존 — 디스크 쓰기 + WAL + fsync, 복제 시스템에서는 다른 노드로의 복제 디스크·펌웨어 버그, 백업 손상, 전 replica 동시 장애를 생각하면 완벽한 durability는 존재하지 않는다 — 기법을 겹쳐 위험을 줄일 뿐

Single-object vs Multi-object 연산

  • Single-object: 하나의 객체 쓰기도 원자성이 필요하다 (20KB JSON을 쓰다 전원이 나가면?). storage engine은 crash 복구용 log와 객체별 lock으로 단일 객체 수준을 보장하고, increment, compare-and-set 같은 단일 객체 원자 연산도 제공한다 — 하지만 이것은 진짜 의미의 "트랜잭션"이 아니다.
  • Multi-object transaction이 필요한 이유: foreign key 참조의 유효성, denormalization된 데이터의 동반 갱신, secondary index와 원본의 동기화(인덱스도 사실상 별도의 객체다). 트랜잭션 없이도 구현은 가능하지만 오류 처리가 극도로 복잡해진다.
  • 오류와 abort 처리 — 재시도의 함정: commit은 성공했는데 확인 응답만 유실되면 재시도로 두 번 실행된다(exactly-once 문제); 과부하가 원인이면 재시도가 상황을 악화(backoff 필요); 일시적(transient) 오류(deadlock, failover)만 재시도 가치가 있고 영구적(permanent) 오류(제약 위반)는 무의미; DB 바깥의 side effect(이메일 발송)는 재시도 시 반복될 수 있다.

Weak Isolation 1 — Read Committed

  • No dirty reads: commit되지 않은 데이터를 읽을 수 없다 — 커밋 전의 부분 상태(이메일은 보이는데 unread counter는 아직)나 곧 abort될 데이터를 보는 것을 방지.
  • No dirty writes: commit되지 않은 데이터를 덮어쓸 수 없다 — 두 트랜잭션의 쓰기가 뒤섞이는 것(중고차 구매: listings는 Bob, invoices는 Alice)을 방지.
  • 구현: dirty write는 row-level write lock(commit/abort까지 보유)으로 막는다. dirty read는 read lock 대신 쓰는 중인 객체의 이전 committed 값을 기억해 두었다가 읽기에는 그 옛 값을 돌려준다 — 읽기가 lock을 잡지 않는다.

Weak Isolation 2 — Snapshot Isolation과 MVCC

Read committed로도 못 막는 anomaly: read skew(nonrepeatable read). Alice의 은행 잔고 예시 — 계좌 두 개에 각 $500(합 $1,000), $100 이체 트랜잭션 실행 도중 두 계좌를 조회하면 한쪽은 이체 전($500), 다른 쪽은 이체 후($400)를 봐서 합이 $900으로 보인다. 각 읽기는 모두 committed 데이터였는데도! 일시적 현상이지만 backup(불일치가 영구화)이나 분석 쿼리·integrity check에서는 치명적이다.

해법 = Snapshot isolation: 각 트랜잭션이 시작 시점에 commit되어 있던 일관된 스냅샷만 읽는다 (PostgreSQL, MySQL InnoDB, Oracle, SQL Server). 구현 기법이 MVCC(multi-version concurrency control)다.

  • 원칙: readers never block writers, writers never block readers.
  • 객체의 여러 버전을 함께 보관: 트랜잭션마다 증가하는 txid, row마다 created_by(만든 txid)와 deleted_by(삭제 표시 txid). update는 delete + create로 변환, 안 보이게 된 버전은 GC가 제거.
  • 인덱스 처리: 단순하게는 인덱스가 모든 버전을 가리키고 조회 시 비가시 버전을 거른다 (PostgreSQL은 같은 페이지 내 버전이면 인덱스 갱신 회피, CouchDB 등은 append-only/copy-on-write B-tree).
  • 이름 혼란: SQL 표준에 snapshot isolation이 없어서 PostgreSQL·MySQL은 repeatable read, Oracle은 serializable이라 부른다(실제론 serializable이 아닌데도).
심층 설명 — snapshot isolation의 MVCC 가시성 규칙: 어떤 트랜잭션에게 어떤 버전이 보이는지는 시계가 아니라 txid의 대수로 결정된다. (1) 내 시작 시점에 이미 commit된 트랜잭션이 만든 버전만 보인다 — 시작 당시 진행 중이던 트랜잭션의 쓰기는 그 뒤에 commit되어도 끝까지 무시한다. (2) abort된 트랜잭션의 쓰기는 무시. (3) 나보다 나중에 시작한 트랜잭션의 쓰기는 commit 여부와 무관하게 무시. (4) 삭제도 같은 규칙 — 나중 트랜잭션이 deleted_by 표시를 해도 내 스냅샷에서는 그 row가 여전히 살아 있다. 이 규칙 덕분에 DB는 아무것도 잠그지 않고도 "그 순간에 세상을 정지시킨" 것 같은 일관된 뷰를 각 트랜잭션에게 제공한다 — 오래 걸리는 backup·분석 쿼리가 쓰기 트래픽과 공존할 수 있는 이유다.

Lost Update 방지

Lost update: 두 트랜잭션이 동시에 read-modify-write cycle을 수행하면 나중 쓰기가 먼저 쓰기를 포함하지 않은 채 덮어써 한쪽의 수정이 사라진다 (counter 증가, 위키 동시 편집, JSON 문서 일부 수정).

해법 내용 주의점
Atomic write operations UPDATE counters SET value = value + 1 ... — cycle 자체를 제거. 가능하면 항상 최선 ORM이 무심코 read-modify-write 코드를 만들기 쉽다
Explicit locking SELECT ... FOR UPDATE로 읽을 row에 lock lock을 잊으면 race condition — 실수하기 쉽다
자동 감지 snapshot isolation과 결합, lost update 감지 시 abort + 재시도 PostgreSQL·Oracle·SQL Server는 지원; MySQL InnoDB repeatable read는 감지 안 함
Compare-and-set UPDATE ... WHERE content = '읽었던 값' — 값이 그대로일 때만 갱신 WHERE절이 옛 스냅샷을 보면 무용지물일 수 있음
복제 환경의 충돌 해결 sibling 보존 + 병합; counter처럼 commutative한 연산은 CRDT가 자동 병합 lock·compare-and-set은 통하지 않음(최신 복사본이 하나가 아님); LWW는 lost update를 일으키는데도 많은 DB의 기본값

Write Skew와 Phantom

Write skew — lost update의 일반화: 두 트랜잭션이 같은 객체들을 읽고, 서로 다른 객체를 갱신할 때 발생한다.

  • 당직 의사 예시: "최소 1명은 on-call" 규칙 아래 Alice와 Bob이 동시에 on-call 수를 조회(둘 다 2명 확인) → 각자 자기 자신의 레코드만 off-call로 갱신 → 둘 다 commit → on-call 0명. snapshot isolation도 못 막는다.
  • 다른 예: 회의실 예약(겹침 확인 후 삽입), 멀티플레이어 게임(다른 말을 같은 자리로), username 선점(중복 확인 후 생성), 이중 인출.
  • 패턴: ① SELECT로 조건 확인 → ② 결과에 따라 ③ 쓰기 — 그 쓰기가 ①의 전제를 바꾼다. 다른 트랜잭션의 쓰기가 내 검색 결과를 바꾸는 효과가 phantom이다.
  • 대응: 진짜 해법은 serializable isolation; 잠글 row가 있으면 FOR UPDATE; row가 없으면 materializing conflicts(충돌을 표현하는 row — 회의실×15분 슬롯 테이블 — 를 미리 만들어 lock 대상으로 삼기)가 있지만 최후의 수단이다.
심층 설명 — write skew가 lost update와 다른 이유, 그리고 phantom: lost update는 두 트랜잭션이 같은 객체를 놓고 싸우는 것이라 "같은 객체에 대한 동시 read-modify-write"만 감지하면 잡을 수 있다(자동 감지, compare-and-set). write skew에서는 두 트랜잭션이 건드리는 객체가 서로 다르다 — Alice는 Alice의 row, Bob은 Bob의 row. 객체 단위로 보면 충돌이 전혀 없다. 무너진 것은 두 쓰기의 공동 전제("on-call ≥ 1")다. 더 나쁜 경우가 phantom: 회의실 예약에서 확인한 전제는 "겹치는 예약이 없다"인데, 존재하지 않는 row에는 lock을 걸 수도, 변경 감지를 할 수도 없다. 그래서 weak isolation 위의 어떤 객체 단위 트릭으로도 근본 해결이 안 되고, serializability(2PL의 predicate/index-range lock, 또는 SSI의 premise 추적)가 필요해진다. materializing conflicts는 "없는 row" 문제를 "미리 만들어둔 row"로 바꾸는 편법 — 동시성 제어가 데이터 모델로 새어 나오므로 최후의 수단이다.

Serializability 구현 1 — Actual Serial Execution

  • 동시성을 아예 제거: 트랜잭션을 단일 스레드에서 하나씩 실행 (Redis, VoltDB/H-Store). 배경: RAM이 싸져서 전체 데이터를 메모리에 올릴 수 있고, OLTP 트랜잭션은 짧고 작다.
  • Stored procedure 필수: 대화형 다중 문 트랜잭션은 네트워크 왕복 동안 단일 스레드가 놀게 되므로, 트랜잭션 전체 코드를 미리 제출해 한 번에 실행. 단점 — DB별 사투리, 디버깅·버전 관리·테스트 어려움, 잘못 짠 프로시저가 DB 전체를 마비 (현대 시스템은 일반 언어: VoltDB는 Java, Redis는 Lua).
  • Partitioning의 한계: 파티션별 단일 스레드로 쓰기 확장이 가능하지만, 여러 파티션에 걸치는 트랜잭션은 파티션 간 조율 때문에 극적으로 느리다(VoltDB 기준 초당 약 1,000건). 단일 파티션 워크로드에만 적합.

Serializability 구현 2 — Two-Phase Locking (2PL)

  • 수십 년간 사실상 유일한 구현 (MySQL InnoDB·SQL Server의 serializable, DB2). 비관적(pessimistic) 동시성 제어.
  • 객체마다 shared lock(읽기, 공유 가능)과 exclusive lock(쓰기, 독점) — 그리고 트랜잭션 끝까지 보유(획득 단계 + 해제 단계 = two-phase). readers block writers, writers block readers — MVCC의 mantra와 정반대.
  • 성능 비용: lock 오버헤드 + 대기 — 느린 트랜잭션 하나가 뒤를 다 세워 지연이 불안정하고, deadlock이 잦다(DB가 감지해 한쪽 abort).
  • Phantom 대응: predicate lock(검색 조건 자체에 lock — 아직 없는 객체까지 커버, 대신 매칭 검사가 비쌈) 또는 그 근사인 index-range locking(next-key locking)(조건을 포함하는 더 넓은 인덱스 범위를 잠금 — 정밀도는 낮지만 오버헤드가 작아 대부분의 2PL DB가 채택).

Serializability 구현 3 — Serializable Snapshot Isolation (SSI)

  • 2008년 등장한 낙관적(optimistic) 알고리즘 — PostgreSQL 9.1+의 serializable, FoundationDB. snapshot isolation 기반으로 일단 실행시키고, commit 시점에 격리 위반을 검사해 위반이면 abort.
  • write skew의 구조 — 읽은 전제(premise)가 commit 시점엔 더 이상 참이 아닐 수 있다 — 를 두 경로로 감지: (1) stale한 MVCC 읽기(내가 무시했던 미commit 쓰기가 먼저 commit됨 → 내 commit 시 abort), (2) 과거 읽기에 영향을 주는 쓰기(누가 어떤 데이터를 읽었는지 기록해 두고, 쓰기 발생 시 최근에 그 데이터를 읽은 트랜잭션들에게 통지).
  • 약점: abort 비율 — 읽고-쓰는 트랜잭션이 길수록 충돌·abort 확률이 커져, 짧은 read-write 트랜잭션을 요구한다.
심층 설명 — 2PL vs SSI 트레이드오프: 둘 다 완전한 serializability를 주지만 비용을 내는 시점이 다르다. 2PL은 비관적 — 문제가 될 소지가 있으면 일단 세운다(lock 대기). 충돌이 실제로 잦은 워크로드에서는 헛수고가 없지만, 읽기와 쓰기가 서로를 블로킹하므로 느린 트랜잭션 하나가 전체 지연을 끌어올리고 deadlock 처리 비용도 낸다. SSI는 낙관적 — 아무도 기다리지 않고 실행한 뒤 commit에서 정산한다. 트랜잭션 간 블로킹이 없어 지연이 예측 가능하고 읽기 전용 쿼리는 lock 없이 스냅샷 위에서 달리며, actual serial execution과 달리 단일 코어에 묶이지 않고 여러 파티션으로 확장할 수 있다. 대신 contention이 높으면 abort로 버려지는 일이 많아진다 — 같은 데이터를 놓고 다투는 긴 read-write 트랜잭션이 많다면 2PL이, 충돌이 드물고 짧은 트랜잭션 위주라면 SSI가 유리하다.

쉬운 예시 / 비유

Atomicity — 계좌 이체 취소 버튼

이체는 "출금 + 입금" 두 동작이지만 하나로 취급된다. 중간에 정전이 나면 출금만 된 상태로 남는 게 아니라 없던 일이 된다. 원자성은 "동시에"가 아니라 "전부 아니면 전무 + 취소 가능"이다.

Consistency — 가계부 검산

"수입 − 지출 = 잔액"이라는 규칙은 가계부 앱(DB)이 아니라 쓰는 사람(애플리케이션)이 지켜야 한다. 앱은 적힌 대로 저장할 뿐, 엉터리로 적으면 엉터리로 저장된다.

Dirty read — 보내다 만 이메일

친구가 편지를 쓰는 중간에 어깨너머로 읽으면(dirty read) 결론이 뒤집힌 문장을 오해한다. 다 쓰고 봉투에 넣은(commit) 편지만 읽어야 한다.

Read skew — 이사 중인 집 사진 찍기

이삿짐이 옮겨지는 도중 옛집과 새집을 시간차를 두고 찍으면, 소파가 양쪽 어디에도 없는 사진 세트가 나온다. snapshot isolation은 "한 순간에 모든 방을 동시에 찍은" 사진을 준다 — backup이 꼭 이런 사진이어야 한다.

MVCC — 문서의 버전 기록

문서를 고칠 때 덮어쓰지 않고 "v3(작성: 트랜잭션 12, 삭제: 트랜잭션 13)"처럼 버전을 쌓아둔다. 독자는 자기가 읽기 시작한 시점에 확정돼 있던 버전만 본다 — 작가와 독자가 서로 기다릴 필요가 없는 이유.

Lost update — 위키 동시 편집

두 사람이 같은 문단을 복사해 각자 고친 뒤 차례로 붙여넣으면, 나중에 붙여넣은 사람이 앞사람의 수정을 통째로 날린다. 해법은 "문단 잠그고 고치기(lock)" 또는 "붙여넣기 전 원본이 그대로인지 확인(compare-and-set)".

Write skew — 당직 맞교대의 함정

의사 둘이 동시에 당직표를 보고 "나 말고도 한 명 있네" 하고 각자 빠진다 → 당직 0명. 서로 다른 줄(자기 이름)만 고쳤기 때문에 lost update 감지에도 안 걸린다. 쓰기가 겹친 게 아니라 서로의 전제를 무너뜨린 것이 문제.

Phantom — 없는 자리에는 자물쇠를 못 건다

회의실 예약에서 잠가야 할 것은 "겹치는 예약이 없다는 사실"인데, 없는 row에는 lock을 걸 수 없다. materializing conflicts는 시간 슬롯이라는 가짜 좌석표를 미리 만들어 거기에 자물쇠를 거는 편법이다.

Actual serial execution — 창구 하나인 은행

창구가 하나면 줄이 생기지만 새치기 문제(동시성 버그)는 원천 차단된다. 단, 손님이 창구 앞에서 전화 통화(대화형 쿼리)를 하면 안 되고, 용건을 미리 쪽지에 적어(stored procedure) 와야 한다.

2PL vs SSI — 검문 vs 사후 심사

2PL은 출입문마다 검문(lock)해서 충돌을 사전에 막는다 — 안전하지만 줄이 길다. SSI는 자유롭게 통과시키고 출국 심사(commit) 때 문제가 발견되면 되돌려보낸다(abort) — 보통은 빠르지만, 위반자가 많으면(높은 contention) 헛걸음이 많아진다.

FOR UPDATE 읽는 row에 명시적 lock을 거는 SQL 절

Phantom 한 트랜잭션의 쓰기가 다른 트랜잭션의 검색 결과를 바꾸는 효과

Actual serial execution 단일 스레드로 트랜잭션을 하나씩 실행 (Redis, VoltDB)

Shared / Exclusive lock 여럿이 공유하는 읽기 lock / 독점하는 쓰기 lock

한 줄 요약

트랜잭션은 온갖 fault와 동시성 문제를 abort 하나로 단순화하는 추상화다 — 하지만 ACID의 I(격리)는 실무에서 대부분 약하다(read committed는 dirty read/write만, snapshot isolation(MVCC)도 lost update 일부와 write skew·phantom은 못 막는다). 완전한 serializability는 단일 스레드 실행, 2PL(비관적), SSI(낙관적) 셋 중 하나로 구현한다.
