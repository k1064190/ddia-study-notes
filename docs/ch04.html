Ch4. Encoding and Evolution

코드와 데이터는 동시에 바뀌지 않는다 — backward/forward compatibility를 지키는 encoding 포맷과, 데이터가 흐르는 세 가지 경로(DB, service call, message)

챕터 개요

애플리케이션은 시간이 지나면 반드시 바뀐다. 기능이 추가되고 데이터 구조(schema)가 변한다. 문제의 핵심은 코드와 데이터가 한순간에 동시에 바뀌지 않는다는 점이다.

  • 서버: 보통 rolling upgrade(노드를 몇 개씩 순차 교체하는 무중단 배포)로 배포한다. 그동안 잠깐 신버전과 구버전 코드가 공존한다.
  • 클라이언트: 사용자가 앱 업데이트를 안 할 수도 있다. 구버전 클라이언트가 한동안 남는다.

그래서 데이터를 주고받을 때 두 방향의 호환성이 필요하다.

호환성 정의 난이도
Backward compatibility (하위 호환) 새 코드가 옛 코드가 쓴 데이터를 읽을 수 있다 보통 어렵지 않음 — 새 코드는 옛 포맷을 이미 알고 있다
Forward compatibility (상위 호환) 옛 코드가 새 코드가 쓴 데이터를 읽을 수 있다 더 어렵다 — 옛 코드는 미래의 추가분을 모르므로, 모르는 부분을 무시할 수 있어야 한다
심층 설명 — forward compatibility가 더 어려운 이유는 "지식의 비대칭" 때문이다. backward compat에서는 새 코드가 과거를 알고 있으므로 옛 포맷 처리 로직을 명시적으로 넣을 수 있다. 반면 forward compat에서는 옛 코드가 미래를 예측할 수 없으므로, 포맷 자체가 "모르는 데이터를 만나면 안전하게 건너뛰고, 필요하면 보존한다"는 규칙을 미리 내장해야 한다. 실무적 함의: 새 field를 추가하기 전에, 이미 배포된 구버전 코드가 그 field를 어떻게 다룰지가 결정되어 있어야 한다는 뜻이다. 흔한 오해는 "서버만 먼저 올리면 forward compat이 필요 없다"는 것인데, rolling upgrade 중이나 구버전 클라이언트가 남아 있는 한 옛 코드는 언제든 새 데이터를 만날 수 있다.

이 챕터는 (1) 데이터를 바이트로 바꾸는 encoding 포맷들과 그것들이 schema 변화를 얼마나 잘 견디는지, (2) 데이터가 프로세스 사이를 흐르는 세 가지 방식(database, service call, message)을 다룬다.

핵심 개념 정리

Encoding이란? (Serialization)

데이터는 두 가지 표현 사이를 오간다.

  • In-memory 표현: 객체, struct, list, array, hash table 등. 포인터로 연결되어 CPU가 다루기 좋다.
  • Byte sequence 표현: 파일 저장이나 네트워크 전송을 위한 연속된 바이트. 자기완결적이며 포인터가 없다.
방향 이름
in-memory → bytes Encoding = serialization = marshalling
bytes → in-memory Decoding = parsing = deserialization = unmarshalling

언어 전용 포맷의 함정

Java Serializable, Python pickle, Ruby Marshal 등 언어 내장 직렬화는 편하지만 위험하다.

  • 특정 언어에 묶인다 — 다른 언어가 읽기 어렵다.
  • 보안 문제 — 디코딩 시 임의 클래스를 인스턴스화할 수 있어 공격에 악용된다.
  • 버전 관리(호환성)가 빈약하고, 효율도 나쁜 경우가 많다.

→ 일시적인 용도 외에는 쓰지 말 것.

JSON, XML, CSV (텍스트 포맷)

널리 쓰이고 사람이 읽을 수 있다. 하지만 약점이 있다.

  • 숫자 모호성: 정수/부동소수 구분이 불명확하다. 특히 JSON은 253보다 큰 정수에서 정밀도가 깨진다 (예: Twitter의 tweet ID가 깨져서 문자열로도 함께 보내야 했던 사례).
  • Binary string 미지원: 바이너리 데이터를 직접 못 담아 Base64로 우회한다 (크기 약 33% 증가).
  • Schema 지원이 선택적: XML Schema / JSON Schema는 강력하지만 복잡하고 잘 안 쓴다. CSV는 schema가 아예 없어 의미가 모호하다.

→ 결함은 있어도 조직 간 데이터 교환 같은 용도엔 충분히 좋다("good enough").

Binary encoding — JSON의 binary 변형

조직 내부에서만 쓴다면 더 작고 빠른 binary 포맷이 낫다. JSON의 binary 변형으로 MessagePack, BSON 등이 있지만, 이들은 여전히 field 이름을 데이터에 포함하기 때문에 절감 효과가 제한적이다.

Thrift와 Protocol Buffers (schema 기반)

둘 다 schema(IDL)를 먼저 정의하고 코드 생성을 한다.

  • 핵심 아이디어: 각 field에 tag number(field tag)와 타입을 부여한다. field 이름은 인코딩된 데이터에 들어가지 않고 tag 숫자만 들어간다 → 매우 compact.
  • Thrift에는 BinaryProtocol과 CompactProtocol(가변 길이 정수, zigzag 인코딩으로 더 작게)이 있다. Protobuf도 유사한 방식이다.

Schema evolution 규칙 — tag가 호환성의 열쇠:

변경 규칙 호환성 근거
Field 추가 새 tag number를 부여. 추가 field는 optional이거나 default 값이 있어야 함 (나중에 추가한 field를 required로 만들 수 없음) 옛 코드는 모르는 tag를 무시 → forward compat. 새 코드는 없는 field를 optional/default로 처리 → backward compat
Field 삭제 optional field만 삭제 가능. 삭제한 tag number는 절대 재사용 금지 재사용하면 옛 데이터와 충돌
타입 변경 가능은 하나 주의 정밀도 손실 / truncation 위험
심층 설명 — tag number 재사용이 절대 금지인 이유는, 인코딩된 데이터에 남는 정보가 오직 "tag 숫자 + 타입 + 값"뿐이기 때문이다. 예컨대 tag 3이 과거 phone_number(string)였다가 삭제되고, 나중에 tag 3을 score(int)로 재사용하면, 디스크나 로그에 남아 있는 옛 레코드의 tag 3 데이터를 새 코드가 score로 잘못 해석한다 — 파싱 에러면 그나마 다행이고, 타입이 우연히 맞으면 조용히 잘못된 값이 들어간다. 데이터는 코드보다 오래 살기 때문에(data outlives code) "그 tag를 쓴 데이터가 이제 없다"고 확신할 방법이 사실상 없다. 실무에서는 Protobuf의 reserved 키워드로 삭제한 tag를 영구 봉인하는 것이 관례다.

Avro

Thrift/Protobuf와 달리 tag number가 없다. 대신 두 개의 schema를 쓴다.

  • Writer's schema(데이터를 쓸 때의 schema) + reader's schema(읽을 때의 schema).
  • Schema resolution: 읽을 때 두 schema를 비교해 field 이름으로 매칭한다. 둘이 똑같을 필요는 없고 호환되기만 하면 된다.
  • 데이터에 field 이름도 tag도 없어 가장 compact하다 (오로지 schema에 의존).

Reader는 writer's schema를 어떻게 아는가? (상황별)

상황 방법
큰 파일에 많은 레코드 (예: Hadoop) 파일 맨 앞에 writer's schema를 한 번 넣는다 (Avro object container file)
DB에 레코드를 개별 기록 레코드마다 schema 버전 번호를 붙이고, 버전→schema 매핑은 schema registry에 둔다
네트워크 전송 연결 수립 시 schema를 협상(negotiate) (Avro RPC)

Avro의 강점 — 동적 생성 schema에 친화적: tag number를 사람이 일일이 배정할 필요가 없어, DB schema에서 Avro schema를 자동 생성하기 좋다 (예: DB를 통째로 파일로 덤프). 코드 생성도 선택 사항이라 동적 타입 언어에 잘 맞는다.

심층 설명 — Avro의 "tag 없는" 인코딩이 성립하는 메커니즘: 인코딩된 바이트에는 field 식별자가 전혀 없고 값들만 schema에 정의된 순서대로 이어져 있다. 따라서 디코딩하려면 그 데이터를 쓴 시점의 writer's schema를 바이트 단위로 정확히 알아야 하고, reader's schema와 다르면 schema resolution이 이름 기준으로 번역해 준다 — writer에만 있는 field는 건너뛰고, reader에만 있는 field는 default 값으로 채운다. 실무적 함의: writer's schema를 전달하는 채널(container file 헤더, schema registry, 협상)이 시스템 설계의 필수 요소가 된다. 흔한 오해는 "schema가 없어도 되는 포맷"이라는 것 — 정반대로, Avro는 schema에 가장 강하게 의존하는 포맷이며, 그 대가로 최고의 compact함과 동적 schema 생성 친화성을 얻는다.

Schema 기반 binary 인코딩의 장점

JSON/XML 대비:

  • field 이름을 빼서 더 compact하다.
  • schema 자체가 항상 최신인 문서 역할을 한다.
  • 배포 전에 schema 호환성을 자동 검사할 수 있다 (특히 schema registry).
  • 코드 생성 → 컴파일 타임 타입 체크.

Modes of Dataflow ① — 데이터베이스를 통해

한 프로세스가 DB에 쓰고, 다른(또는 나중의 같은) 프로세스가 읽는다.

  • Backward compat은 당연히 필요하다 (새 코드가 옛 데이터를 읽는다).
  • Forward compat도 필요하다: rolling upgrade 중엔 옛 코드가 새 코드가 쓴 데이터를 읽을 수 있다.
  • 함정 — 모르는 field 보존: 옛 코드가 새 field가 포함된 레코드를 읽어서 갱신·저장할 때, 자기가 모르는 field를 버리면 안 된다. 모르는 field를 유지한 채로 다시 써야 한다.
  • 데이터는 코드보다 오래 산다 (data outlives code): 옛 데이터를 통째로 마이그레이션하는 건 비싸므로, schema evolution 덕분에 옛 포맷 데이터를 그대로 둘 수 있다.

Modes of Dataflow ② — 서비스 호출을 통해 (REST와 RPC)

클라이언트–서버 구조. microservice/SOA에서는 서비스끼리 서로 호출한다.

  • REST: HTTP 위의 설계 철학 (resource, URL, HTTP 동사 활용). 단순하고 널리 쓰인다.
  • SOAP: XML 기반 (WSDL 등). 복잡하고 요즘은 선호도가 낮다.
  • RPC (Remote Procedure Call): 네트워크 호출을 로컬 함수 호출처럼 보이게 하려는 시도. 하지만 이 추상화는 새는 추상화(leaky abstraction)다:
    • 네트워크는 불안정하다 — timeout, 실패 가능.
    • 지연(latency)이 있고 변동한다.
    • 재시도 시 idempotence 문제가 있다 (같은 요청이 두 번 처리될 수 있음).
    • 큰 인자 전달과 언어 간 타입 변환이 까다롭다.
    → RPC ≠ 로컬 호출.
  • 현대 RPC 프레임워크: gRPC(Protobuf), Finagle(Thrift), Rest.li(JSON/HTTP). future·stream·service discovery를 지원한다.
  • 호환성: 보통 서버를 먼저 업그레이드하므로, 요청에는 backward compat(새 서버가 옛 클라이언트의 요청을 읽음), 응답에는 forward compat(옛 클라이언트가 새 서버의 응답을 읽음)이 필요하다. API 버전 관리로 다룬다.

Modes of Dataflow ③ — 비동기 메시지 전달을 통해

Message broker / message queue를 거친다. RPC(저지연)와 DB(durable)의 중간 성격이다. 송신자가 broker의 queue/topic에 메시지를 보내면, broker가 consumer에게 전달한다.

  • 수신자가 죽어도 버퍼링해 두었다가 재전달 → 안정성 향상.
  • 송신자가 수신자의 IP/위치를 몰라도 됨 → decoupling.
  • 한 메시지를 여러 수신자에게 전달 가능.
  • 송신자와 수신자가 서로 독립적으로 배포·확장 가능.

Broker 예: RabbitMQ, ActiveMQ, Kafka 등. 메시지는 결국 바이트 + 메타데이터이므로, encoding의 forward/backward compat이 있으면 publisher와 consumer를 독립적으로 업데이트할 수 있다.

Distributed actor framework: actor model(상태를 가진 actor들이 메시지로만 소통 — thread/lock 없이 동시성 처리)을 여러 노드로 확장한 것. Location transparency(같은 머신이든 네트워크 너머든 동일하게 메시지 전달)가 핵심 성질이다. 예: Akka, Orleans, Erlang OTP. rolling upgrade 시에는 여기서도 메시지 encoding 호환성이 필요하다.

쉬운 예시 / 비유

Backward vs Forward compatibility — Word 문서

새 Word로 옛 .doc을 여는 건 backward compat(보통 잘 된다). 옛 Word로 새 기능이 든 문서를 여는 게 forward compat — 옛 Word는 모르는 새 기능을 무시하고 나머지는 보여줄 수 있어야 한다. 후자가 더 어렵다.

Field tag — 번호표 붙은 서랍

옷장 서랍에 "이름표" 대신 번호표(1번, 2번…)를 붙이는 것. 데이터엔 번호만 적고, 무엇이 들었는지는 schema(번호↔의미 표)를 보면 안다. 이름을 매번 안 적으니 짐이 가볍다. 단, 1번 서랍을 비웠다고 나중에 1번을 다른 용도로 재사용하면 옛 기록과 꼬인다 (= tag 재사용 금지).

모르는 field 무시 — forward compat의 태도

옛 코드가 새 데이터를 읽을 때, 처음 보는 항목은 "이건 내가 모르는 거니 건드리지 말고 그대로 두자"라고 처리한다. 그래야 신·구 버전이 공존할 수 있다.

모르는 field 보존 — DB dataflow의 함정

옛 버전 앱이 사용자 프로필을 수정해 저장할 때, 신버전이 추가한 "취미" 항목을 모른다고 지워버리면 안 된다 — 모른 채로 같이 보존해서 다시 써야 한다.

RPC ≠ 로컬 호출 — 옆자리 대화 vs 국제 우편

옆자리 동료에게 말 거는 것(로컬 함수)과, 외국에 있는 사람에게 편지를 보내는 것(RPC)은 다르다. 편지는 분실·지연될 수 있고, 같은 편지를 두 번 보내면 상대가 두 번 처리할 수도 있다(idempotence 문제).

Message broker — 우체국

보내는 사람은 받는 사람이 지금 집에 있는지 몰라도 된다. 우체국이 보관했다가 전달하고, 같은 안내문을 여러 집에 돌릴 수도 있다.
