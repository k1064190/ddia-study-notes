Ch6. Partitioning

큰 데이터셋과 높은 부하를 여러 노드에 골고루 나누기 — key range vs hash, local vs global index, rebalancing, 그리고 요청을 올바른 partition으로 보내는 routing

챕터 개요

데이터셋이 너무 크거나 query 처리량이 너무 높으면 한 노드로는 감당할 수 없다. 해결책은 데이터를 partition(shard)으로 쪼개 여러 노드에 나누어 놓는 것 — scalability가 목적이다.

  • 각 데이터 조각(record, row, document)은 정확히 하나의 partition에 속한다.
  • 큰 데이터셋을 여러 디스크에, query 부하를 여러 프로세서에 분산할 수 있다.
  • Partitioning은 보통 replication과 함께 쓰인다: 각 partition의 복사본을 여러 노드에 둔다. 한 노드는 어떤 partition들의 leader이면서 동시에 다른 partition들의 follower일 수 있다.

핵심 목표는 데이터와 query 부하를 노드들에 골고루 분산하는 것이다. 분산이 불공평해서 일부 partition에 데이터·부하가 몰린 상태를 skew라 하고, 불균형하게 높은 부하를 받는 partition을 hot spot이라 한다.

핵심 개념 정리

Key-value 데이터의 partitioning ① — key range 기준

각 partition에 연속된 key 구간(range)을 할당한다 (백과사전의 A–B권, C–D권처럼).

  • key가 정렬된 상태로 유지되므로 range scan이 쉽다 (예: Bigtable, HBase, RethinkDB, MongoDB의 range 방식).
  • 구간 경계는 데이터 분포에 맞춰 조정해야 한다 (알파벳 균등 분할이 아니라 데이터 양 균등 분할).
  • 약점 — hot spot 위험: 특정 접근 패턴이 한 구간에 몰릴 수 있다. 예: timestamp를 key로 쓰는 sensor 데이터 — 오늘 쓰는 데이터가 전부 "오늘" partition에 몰려, 쓰기 부하가 한 partition에 집중된다.
    • 완화책: key 앞에 sensor 이름을 prefix로 붙여 (sensor별 → 시간순) 쓰기를 분산한다. 단, 특정 시간 범위의 전체 sensor 조회는 sensor별로 여러 range query를 날려야 한다.

Key-value 데이터의 partitioning ② — key의 hash 기준

hash function으로 key를 숫자로 바꾸고, hash 값 구간을 partition에 할당한다.

  • 좋은 hash는 skew된 key도 균등하게 분산시킨다 (Cassandra의 MD5, MongoDB의 hashed sharding 등). 암호학적으로 강할 필요는 없다.
  • 대가 — range query 효율 상실: 인접했던 key들이 hash 후 흩어지므로, range query는 모든 partition에 보내야 한다 (Riak·Couchbase·Voldemort는 primary key range query를 지원하지 않음).
  • 참고: 이 방식은 consistent hashing이라 불리기도 하지만, 책은 이 용어가 (CDN 유래의 특정 기법을 가리켜) 혼란스러우니 hash partitioning이라 부르길 권한다.
  • Cassandra의 절충 — compound primary key: 여러 column 중 첫 column만 hash해서 partition을 정하고, 나머지 column들은 SSTable 안에서 정렬에 쓴다. → 첫 column을 고정하면 나머지 column에 대해 효율적인 range scan이 가능 (예: (user_id, update_timestamp) — 한 사용자의 최근 게시물을 시간순으로 조회).
기준 Key range partitioning Hash partitioning
정렬 key 정렬 유지 정렬 상실 (인접 key가 흩어짐)
range scan 효율적 (한/소수 partition) 모든 partition에 query 필요
부하 분산 접근 패턴에 따라 hot spot 위험 균등 분산에 유리
대표 시스템 Bigtable, HBase, RethinkDB Cassandra, MongoDB(hashed), Riak, Voldemort
심층 설명 — key range vs hash는 "정렬을 지키느냐, 분산을 지키느냐"의 trade-off다. key range는 데이터의 의미적 인접성을 물리적 인접성으로 보존하므로 range scan이 한 partition 안에서 끝나지만, 그 인접성 때문에 접근 패턴이 한 구간에 몰리면(예: timestamp key의 "오늘" 구간) 쓰기 hot spot이 생긴다. hash는 인접성을 일부러 파괴해서 부하를 확률적으로 고르게 만들지만, 그 대가로 range query가 전 partition scatter가 된다. 그래서 실전 시스템은 중간 지점을 찾는다 — Cassandra의 compound key는 "partition 간에는 hash로 분산, partition 안에서는 정렬 유지"라는 2단 구조로 두 세계를 절충한다. 핵심은 "어떤 축으로 몰리고, 어떤 축으로 훑는가"를 파악해 몰리는 축을 hash하고 훑는 축을 정렬에 남기는 것이다.

Skewed workload와 hot spot 완화

hash partitioning도 극단적으로 skew된 workload는 못 막는다 — 모두가 같은 key를 읽고 쓰면 (예: 팔로워 수백만 celebrity의 user ID에 이벤트가 폭주) 그 key의 partition이 hot spot이 된다.

  • 현재 대부분의 시스템은 이를 자동으로 보정하지 못한다 → application의 책임이다.
  • 기법: 극소수의 hot key에만 random suffix(예: 두 자리 숫자 00~99)를 붙여 100개 key로 쪼개 여러 partition에 분산한다.
    • 대가: 읽기는 100개 key를 전부 읽어 합쳐야 하고, 어떤 key를 쪼갰는지 추적하는 부가 관리가 필요하다. 그래서 소수의 hot key에만 적용해야 한다.

Partitioning과 secondary index ① — document 기반 (local index)

  • 각 partition이 자기 partition 안의 document만 대상으로 secondary index를 유지한다 → local index. 쓰기는 해당 document가 있는 partition만 갱신하면 된다.
  • 읽기의 대가 — scatter/gather: secondary index로 검색하려면 (원하는 값이 어느 partition에 있을지 모르므로) 모든 partition에 query를 보내고 결과를 합쳐야 한다. 병렬로 보내도 tail latency amplification(가장 느린 partition이 전체 응답을 좌우)에 취약하다.
  • 그래도 널리 쓰인다: MongoDB, Riak, Cassandra, Elasticsearch, SolrCloud, VoltDB 등.

Partitioning과 secondary index ② — term 기반 (global index)

  • 모든 partition의 데이터를 아우르는 global index를 만들되, 이 index 자체도 한 노드에 두면 병목이므로 term(색인하는 값) 기준으로 partition한다. (예: color:red는 index partition 0에, color:silver는 partition 1에.)
  • term 자체로 나누면 range scan에 유리, term의 hash로 나누면 부하 분산에 유리.
  • 읽기가 효율적: 원하는 term이 있는 index partition 하나만 조회하면 된다 (scatter/gather 불필요).
  • 쓰기의 대가: 한 document 쓰기가 여러 index partition을 건드릴 수 있다 → 동기 갱신에는 분산 트랜잭션이 필요한데 대부분 지원하지 않아, 실무에서는 global index를 비동기(asynchronous)로 갱신한다 (쓰기 직후 index에 바로 반영되지 않을 수 있음 — 예: Amazon DynamoDB의 GSI).
심층 설명 — local vs global index는 "비용을 쓰기에 내느냐, 읽기에 내느냐"의 선택이다. local index는 쓰기가 document의 partition 안에서 완결되므로 빠르고 트랜잭션적으로 단순하지만, 읽기가 전 partition scatter/gather가 되어 partition 수가 늘수록 검색 비용과 tail latency가 함께 커진다. global index는 그 비용을 쓰기 쪽으로 옮긴다 — 한 document의 term들이 여러 index partition에 흩어져 있으니 쓰기 한 건이 다중 partition 갱신이 되고, 이를 동기로 하려면 분산 트랜잭션이 필요하다. 대부분의 시스템이 이를 포기하고 비동기 갱신을 택하므로, global index는 "방금 쓴 데이터가 검색에 안 보일 수 있다"는 일관성 지연을 안고 간다. 즉 read-heavy 검색이면 global, write-heavy이고 검색 fan-out을 감당할 수 있으면 local이 자연스러운 선택이다.

Rebalancing — 하면 안 되는 방법: hash mod N

시간이 지나면 query 증가·데이터 증가·노드 장애로 partition을 노드 간에 옮겨야 한다 = rebalancing. 요구사항: 부하가 공평해질 것, rebalancing 중에도 서비스가 계속될 것, 필요 이상으로 데이터를 옮기지 말 것.

hash(key) mod N은 나쁘다 — 노드 수 N이 바뀌면 대부분의 key가 다른 노드로 재배치된다. 예: hash 123456은 N=10이면 노드 6, N=11이면 노드 3, N=12이면 노드 0.
심층 설명 — mod N이 나쁜 이유는 "N이 수식 안에 박혀 있다"는 것 자체다. 노드 하나만 추가해도 거의 모든 key의 mod 결과가 바뀌므로, 클러스터 전체가 데이터를 서로 주고받는 대이동이 일어난다 — rebalancing의 목표(최소 이동)와 정반대다. 좋은 전략들의 공통 원리는 key→partition 매핑과 partition→노드 배정을 분리하는 것이다. 고정 개수 partition에서는 key→partition은 영원히 불변이고, 노드가 늘면 partition 몇 개만 통째로 새 노드로 옮긴다(이동량 ≈ 1/노드 수). 동적 partitioning과 노드 비례 방식도 마찬가지로 "옮기는 단위"를 partition으로 국한한다. 즉 rebalancing 설계의 핵심 질문은 "노드 수가 바뀔 때 무엇이 불변으로 남는가"이며, mod N은 불변인 것이 하나도 없어서 실격이다.

Rebalancing — 실제 전략들

전략 동작 특징 / 주의점 시스템
고정 개수 partition 노드 수보다 훨씬 많은 partition을 처음부터 생성 (예: 노드 10대에 1,000개). 노드 추가 시 partition 몇 개를 통째로 이동 key→partition 매핑 불변. 개수 선택이 어렵다 — 너무 많으면 관리 오버헤드, 너무 적으면 재분배 여지 없음 Riak, Elasticsearch, Couchbase, Voldemort
동적 partitioning 임계 크기 초과 시 split, 너무 작으면 merge — B-tree와 유사 partition 수가 데이터 총량에 적응. 빈 DB는 partition 1개로 시작해 초기 쓰기가 한 노드에 몰림 → pre-splitting으로 완화 HBase, RethinkDB
노드 수 비례 노드당 partition 수 고정 (Cassandra 기본 256개). 새 노드가 기존 partition들을 랜덤하게 골라 절반씩 가져감 데이터가 커지면 partition도 커지고, 노드를 늘리면 다시 작아짐 Cassandra

Rebalancing — 자동 vs 수동

  • 완전 자동 rebalancing은 편하지만 위험하다: 요청 라우팅 변경 + 대량 데이터 이동이라 비싸며, 잘못하면 네트워크·노드에 과부하를 준다.
  • 최악의 시나리오: 자동 장애 감지와 결합될 때 — 과부하로 느려진 노드를 "죽었다"고 판단해 자동으로 rebalancing을 시작하면, 그 부하가 다른 노드와 네트워크를 더 짓눌러 cascading failure로 번질 수 있다.
  • 그래서 사람이 개입하는 rebalancing이 더 안전한 경우가 많다 (Couchbase·Riak·Voldemort는 재배정을 자동 제안하되 적용은 관리자 승인).

Request routing (service discovery)

클라이언트는 "key foo가 지금 어느 노드에 있는지"를 어떻게 아는가? 세 가지 접근:

# 방식 설명
1 아무 노드에나 접속 담당이면 처리, 아니면 담당 노드로 전달(forward)하고 응답을 중계
2 routing tier 모든 요청을 partition-aware한 라우팅 계층으로 보내고 거기서 담당 노드로 전달 (partition-aware load balancer)
3 partition-aware client 클라이언트가 partition→노드 매핑을 직접 알고 바로 접속
  • 핵심 난제는 라우팅 주체가 partition 배정 변화를 어떻게 최신으로 아느냐다.
  • 많은 시스템은 ZooKeeper 같은 별도 coordination service에 cluster metadata를 두고, 변경 시 라우팅 계층에 통지한다 (HBase, SolrCloud, Kafka; MongoDB는 자체 config server).
  • Cassandra와 Riak은 gossip protocol을 사용 — 노드끼리 cluster 상태 변화를 서로 퍼뜨리고, 아무 노드나 요청을 받아 올바른 노드로 전달한다 (외부 coordination service 불필요, 방식 1).
  • MPP(massively parallel processing) 분석용 DB는 join·filter·group·aggregation을 포함하는 복잡한 query를 여러 stage·partition으로 쪼개 병렬 실행하는 훨씬 정교한 query 최적화를 한다 (자세한 내용은 Ch10).

쉬운 예시 / 비유

Partitioning — 백과사전 분권

한 권에 다 담을 수 없으니 A–B권, C–D권으로 나눈다. "B로 시작하는 단어"를 찾으면 한 권만 보면 된다 (key range partitioning + range scan).

Hot spot — 인기 창구

은행 창구 10개를 열어도 모두가 "1번 창구 직원"만 찾으면 나머지 9개는 놀고 1번만 줄이 늘어선다. 창구를 늘려도(partition 추가) 소용없다 — 손님을 분산시키는 건 결국 운영자(application)의 몫이다 (celebrity key + random suffix).

Key range vs hash — 도서관 vs 물품보관함

도서관은 주제별로 정렬돼 있어 "역사책 구역"을 통째로 훑기 좋다(range scan). 물품보관함은 번호가 무작위로 배정돼 자리가 골고루 차지만, "내 짐들끼리 나란히"는 포기해야 한다.

Local vs global index — 지점별 장부 vs 본사 통합 장부

지점별 장부(local index)는 기록(쓰기)이 그 지점에서 끝나지만, "빨간 차 전부 찾기"는 모든 지점에 전화를 돌려야 한다(scatter/gather). 본사 통합 장부(global index)는 찾을 땐 한 곳만 보면 되지만, 판매(쓰기) 한 건마다 본사의 여러 장부 담당자에게 통지해야 하고, 통지가 늦게 반영될 수 있다(비동기 갱신).

hash mod N — 반 배정을 반 개수로 다시 계산하기

"출석번호 % 반 개수"로 반을 배정하면, 반이 하나 늘어날 때마다 거의 모든 학생이 반을 옮겨야 한다. 좋은 rebalancing은 전학생 몇 명만 이동시킨다 (partition 통째 이동).

자동 rebalancing의 위험 — 과로한 직원을 결근 처리

바빠서 응답이 늦은 직원을 "결근"으로 간주하고 그의 업무를 옆자리로 자동 재배정하면, 옆자리도 과로로 쓰러진다 — cascading failure.

큰 데이터와 높은 부하를 여러 노드에 골고루 나누기 위해 key range 또는 hash로 partition하고(range scan vs 균등 분산의 trade-off), secondary index는 local(쓰기 유리) / global(읽기 유리)로 나뉘며, rebalancing은 필요 최소한의 데이터만 옮기는 전략(고정 개수·동적·노드 비례)을 쓰고, 요청은 forwarding·routing tier·partition-aware client 중 하나로 올바른 partition을 찾아간다.
