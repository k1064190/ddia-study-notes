Ch10. Batch Processing

1970년대 Unix 파이프라인의 철학(한 가지를 잘, 균일한 인터페이스로 조합, 불변 입력)에서 출발해 이를 수천 대로 확장한 MapReduce와 HDFS, 그리고 그 약점(중간 상태 materialization)을 넘어선 dataflow engine·그래프 처리·선언적 API까지

챕터 개요

앞선 장들이 데이터를 어떻게 저장·일관 처리하는가였다면, 이 챕터부터는 데이터를 어떻게 다른 데이터로 변환하는가를 다룬다. Kleppmann은 시스템을 세 종류로 나눈다:

종류 성격 핵심 척도
Service (online) 요청을 받아 가능한 빨리 응답 response time · availability
Batch (offline) 대량 입력으로 작업을 돌려 출력 생성, 주기적 실행 throughput
Stream (near-real-time) 이벤트 발생 즉시 처리하되 파생 출력 생성 batch와 online의 중간 (11장)

이 챕터의 주인공은 batch processing이다. 그 사고방식의 뿌리는 1970년대 Unix 도구와 철학에 있으며, 그것을 분산 규모로 확장한 것이 MapReduce다 — 값싼 commodity 장비 위에서 Google 검색 색인 같은 작업을 무차별 대입으로 돌릴 수 있게 했다.

챕터를 관통하는 핵심 사상은 입력의 불변성(immutability) 이다. 입력을 건드리지 않고 새 출력을 통째로 만들기에 batch 작업은 반복 실행 가능(repeatable)하고, 버그가 있어도 코드를 고쳐 다시 돌리면 되며(human fault tolerance), 롤백이 쉽다.

핵심 개념 정리

Batch Processing with Unix Tools — 사상의 원점

웹 서버 로그에서 "가장 많이 조회된 페이지 5개"를 뽑는 예 — 한 줄짜리 파이프라인으로 풀린다:

cat access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -n 5

각 단계가 한 가지 일만 하고 파이프(|)로 이어진다:

  • 각 프로그램은 한 가지 일을 잘한다(do one thing well) — 새 기능은 새 프로그램을 만들어 조합한다.
  • 구성 가능성(composability): 모든 프로그램의 출력이 다른 프로그램의 입력이 될 수 있다. 이를 가능케 하는 uniform interface는 파일(파일 디스크립터), 구체적으로 stdin/stdout으로 흐르는 text stream(줄바꿈으로 구분된 레코드)이다.
  • logic과 wiring의 분리: 프로그램은 자기 로직만 담고, 입력이 어디서 오고 출력이 어디로 가는지는 shell이 결정한다.
  • sort는 메모리보다 큰 데이터도 디스크로 spill하고 여러 CPU로 병렬 정렬한다 — 이미 "메모리에 안 들어가는 데이터"를 다룬다.
  • 입력의 불변성: 파이프라인은 입력 파일을 결코 수정하지 않는다 — 옵션을 바꿔 몇 번이고 다시 돌려도 안전하고 중간 결과를 들여다보며 자유롭게 실험·디버깅할 수 있다.
심층 설명 — Unix 철학과 MapReduce의 공통점 = 균일한 인터페이스: Unix가 프로그램을 레고처럼 이어 붙일 수 있는 비결은 단순하고 느슨한 공통 인터페이스다 — 모두가 stdin으로 읽고 stdout으로 쓰는 text stream 하나로 통일돼 있어, 어떤 프로그램의 출력이든 다른 프로그램의 입력이 된다. MapReduce도 정확히 같은 발상을 분산 규모로 옮긴 것이다: Unix의 stdin/stdout 자리에 분산 파일시스템(HDFS)의 파일이 들어선다. mapper는 파일을 읽어 key-value를 쓰고, reducer는 그 결과를 읽어 또 파일을 쓴다. 각 작업이 HDFS 위 불변 파일이라는 균일한 인터페이스로 소통하기에, 서로 다른 팀이 만든 작업들을 파이프라인처럼 엮을 수 있다. 차이는 규모와 실행 위치 뿐 — Unix는 단일 머신의 파이프, MapReduce는 수천 대에 흩어진 파일. 이 "logic(각 프로그램)과 wiring(연결)의 분리" + "불변 입력" 조합이 두 시스템이 공유하는 유전자다.

MapReduce와 분산 파일시스템

MapReduce는 Unix 파이프라인과 닮았지만 수천 대 장비에 분산된다. uniform interface는 stdin/stdout 대신 분산 파일시스템 위의 파일이다.

HDFS (Hadoop Distributed File System): shared-nothing 원칙으로 특별한 하드웨어 없이 일반 데이터센터의 commodity 디스크들을 네트워크로 묶는다. 각 장비의 DataNode 데몬과, 어떤 블록이 어느 장비에 있는지 추적하는 중앙 NameNode로 구성되며, 파일은 여러 장비에 복제·분산돼 장비가 죽어도 살아남는다. 전용 하드웨어로 공유 저장소를 제공하는 비싼 NAS/SAN과 대조적으로, 값싼 장비를 그냥 늘려(scale out) 페타바이트급으로 키운다.

MapReduce 작업의 실행:

  • 입력 파일을 레코드 단위로 쪼갠다.
  • Mapper: 각 입력 레코드마다 호출돼 임의 개수의 key-value 쌍을 방출한다. 레코드 간 상태를 유지하지 않는다.
  • (프레임워크) 정렬: 방출된 모든 key-value를 key로 정렬한다.
  • Reducer: 같은 key의 값들을 모아 호출돼 출력을 만든다.
  • 핵심은 mapper와 reducer 사이의 정렬·재분배 단계 = shuffle이다 — mapper 출력을 key로 partition·정렬해 같은 key의 레코드가 같은 reducer로 모이게 한다.
  • map-side data locality: mapper는 가급적 입력 블록이 저장된 장비에서 실행된다. 큰 입력을 옮기는 대신 코드(수 KB의 JAR)를 데이터가 있는 곳으로 보낸다 — "putting the computation near the data".

MapReduce workflows: 현실 문제는 여러 작업을 체인으로 엮어 푼다 — 앞 작업의 출력 디렉터리가 다음 작업의 입력이 된다. MapReduce 자체엔 workflow 개념이 없어, 작업 의존성을 관리(성공 후 다음 실행)하는 workflow scheduler(Oozie, Airflow, Luigi, Azkaban 등)가 필요하다.

Reduce-side joins와 grouping:

  • MapReduce엔 인덱스가 없어 join은 보통 양쪽 데이터셋 전체를 훑는다.
  • Sort-merge join: 두 데이터셋(활동 로그 + 사용자 프로필)을 각각 join key(user ID)로 방출하면 프레임워크가 key로 정렬해, 같은 user ID의 레코드가 reducer에 나란히 모인다 — reducer가 이 "관련 레코드 모으기"로 join한다. secondary sort로 프로필을 활동보다 먼저 오게 하면 깔끔하다.
  • Group-by도 같은 구조 — grouping key로 방출하면 같은 그룹이 한 reducer로 모인다.
  • Skew(hot key): 특정 key에 레코드가 몰리면 그 reducer 하나가 전체를 지연시킨다(straggler). 대응 — skewed/sharded join(hot key를 여러 reducer로 분산, 상대 레코드는 복제), map-side aggregation(combiner) (reducer 전에 mapper 쪽 부분 집계로 전송량 감소).
심층 설명 — reduce-side sort-merge join과 shuffle: MapReduce에는 랜덤 액세스 인덱스가 없어서 join을 "한쪽을 인덱스로 찾아보기" 로 할 수 없다. 대신 정렬의 힘을 빌린다. 활동 로그의 모든 레코드와 프로필 DB의 모든 레코드를 각각 mapper로 읽되, 둘 다 user ID를 key로 방출한다. 그러면 프레임워크의 shuffle이 마법을 부린다 — 모든 방출 레코드를 key로 partition·정렬해 같은 user ID를 가진 레코드들(활동이든 프로필이든)이 하나의 reducer에, 그것도 인접해서 도착하게 만든다. reducer는 그저 연속으로 들어오는 같은 key의 레코드들을 병합(merge)하면 된다 — 이것이 sort-merge join이다. secondary sort를 걸어 각 user ID에서 프로필 레코드가 활동 레코드보다 먼저 오게 하면, reducer는 프로필을 변수에 담아두고 뒤따르는 활동들에 갖다 붙이기만 하면 된다. 즉 join의 본질은 "관련 레코드를 물리적으로 한자리에 모으는 것"이고, 그 "모으기"를 담당하는 것이 정렬 기반 shuffle이다. 인덱스 없이 오직 정렬로 조인을 해내는 것 — 이것이 batch join의 정수다.

Map-side joins (정렬·shuffle 비용을 없애는 경우):

방식 적용 조건 동작
Broadcast hash join 한쪽이 작아 메모리에 들어감 작은 쪽을 해시테이블로 만들어 모든 mapper에 broadcast, 각 mapper가 큰 쪽을 훑으며 조회
Partitioned hash join 두 입력이 같은 방식·같은 개수로 partition됨 대응 partition끼리만 join — 각 mapper가 자기 몫만 메모리에
Map-side merge join 같은 key로 partition + 정렬까지 됨 reduce-side와 같은 병합을 map 단계에서 바로 수행

Batch workflow의 출력: 검색 색인(search index)이나 배치성 key-value store를 만든다. 철학은 불변 파일을 통째로 새로 만들어 놓고(build offline) 서비스가 로드하는 것 — 라이브 DB에 레코드를 하나씩 쓰지 않는다. 그래서:

  • repeatable — 버그가 있어도 고쳐 다시 돌리면 된다.
  • human fault tolerance — 잘못된 출력도 이전 출력으로 쉽게 롤백된다(되돌릴 수 없는 DB 변경과 대조). 코드 실수의 영향을 최소화한다.
  • 부분 실패한 출력은 버리고 재시도하면 되므로 원자성 걱정이 적다.

Beyond MapReduce — MapReduce를 넘어서

MapReduce의 약점 — 중간 상태의 materialization: 여러 작업을 체인으로 엮을 때 각 작업 출력은 분산 FS에 완전히 기록(fully materialize)되어야 다음 작업이 읽는다. 문제 — ① 다음 작업은 앞 작업이 완전히 끝날 때까지 대기 (straggler 하나가 전체를 지연), ② reducer 출력을 다음 mapper가 그대로 다시 읽는 중복 mapper, ③ 임시 데이터를 복제하는 과한 비용. (Unix 파이프는 반대로 중간 결과를 파일로 떨구지 않고 스트리밍한다.)

Dataflow engines (Spark, Tez, Flink): rigid한 map→reduce 2단계 대신 작업 전체를 operator 그래프로 표현해 한 번에 실행한다. 불필요한 정렬·materialization을 건너뛰고, pipelined execution으로 한 operator의 출력을 디스크에 떨구지 않고 다음으로 바로 흘려보낸다. fault tolerance는 중간 상태 복제 대신 재계산으로 — Spark는 데이터의 생성 계보 (RDD lineage)를 추적해 잃은 partition만 입력부터 다시 계산한다(operator가 deterministic이어야 함).

심층 설명 — dataflow engine이 MapReduce보다 빠른 이유 = 중간 상태 materialization 회피: MapReduce 파이프라인의 진짜 병목은 계산이 아니라 작업 사이마다 분산 파일시스템을 왕복하는 것이다. 작업 A의 reducer 출력이 HDFS에 전부 기록되고(게다가 복제까지 돼), 작업 B의 mapper가 그것을 전부 다시 읽어 시작한다. 이 디스크·네트워크 왕복이 반복될수록 비용이 쌓인다. 게다가 B는 A가 마지막 레코드까지 다 끝내야 시작할 수 있어, A의 straggler 하나가 전체 파이프라인을 멈춰 세운다. Dataflow engine은 이 중간 저장 단계를 없앤다 — 작업 전체를 operator 그래프로 보고, 한 operator의 결과를 디스크에 내려놓지 않고 메모리에서 다음 operator로 곧장 흘려보낸다 (pipelined). 또 MapReduce가 매 단계 강제하던 정렬도 정말 필요한 곳에서만 한다. 대신 중간 상태를 복제해 두지 않으니 노드가 죽으면 데이터를 잃는데, 이를 lineage 기반 재계산으로 메운다 — "이 partition은 이런 연산들로 만들어졌다"는 계보만 있으면 잃은 조각을 입력부터 다시 만들면 된다(그래서 operator가 deterministic이어야 한다). 요컨대: 디스크 materialization을 메모리 pipelining으로, 복제 기반 내결함성을 재계산 기반 내결함성으로 바꾼 것이 속도의 핵심이다.

Graph processing — Pregel / BSP: PageRank처럼 그래프를 반복(iterative) 훑는 알고리즘은 MapReduce와 안 맞는다(매 iteration마다 전체 재실행). Pregel 모델(= bulk synchronous parallel, BSP)에서는 각 정점(vertex)이 매 iteration (superstep)마다 다른 정점에 메시지를 보내고, 다음 superstep에서 받은 메시지로 상태를 갱신한다. 정점 상태는 iteration 사이에 메모리에 유지돼 매번 재읽기가 없고, superstep 경계의 checkpoint로 fault tolerance를 얻는다.

고수준 API와 선언적 질의 최적화: MapReduce 손코딩은 장황하다. Hive(SQL 유사), Pig, Spark SQL/DataFrame 등이 흔한 패턴을 짧게 표현하면 엔진이 MapReduce/dataflow 작업으로 컴파일한다. declarative로의 이동 — 무엇을(어떤 join) 원하는지만 적으면 옵티마이저가 실행 전략(broadcast hash join vs sort-merge join 등)을 자동 선택한다(관계형 옵티마이저와 같은 발상). 또 vectorized execution으로 레코드를 한 건씩이 아니라 묶음으로 CPU 캐시·SIMD 친화적으로 처리해 처리량을 끌어올린다.

쉬운 예시 / 비유

Unix 파이프 — 컨베이어 벨트 위의 전문 일꾼들

각 일꾼(프로그램)은 딱 한 가지 작업만 완벽히 하고 결과물을 옆 벨트로 넘긴다. 벨트 규격(text stream)이 표준이라 누구 옆에 누구를 세워도 맞물린다. 새 기능은 기존 일꾼 개조가 아니라 새 일꾼을 줄에 끼워 넣어 얻는다.

불변 입력 — 원본을 손대지 않는 레시피 실험

재료(입력 파일)를 그대로 두고 요리를 새로 만들어 접시에 담는다. 맛이 이상하면 재료는 멀쩡하니 레시피만 고쳐 다시 만들면 된다. 재료 자체를 태워 없앴다면(가변 DB) 돌이킬 수 없다.

HDFS — 도서관 분관 네트워크 vs 초대형 금고(NAS/SAN)

비싼 중앙 금고 하나 대신, 동네 도서관(commodity 장비)마다 책 사본을 나눠 두고 어느 책이 어느 분관에 있는지 적은 대장(NameNode)을 둔다. 분관 하나가 불타도 다른 곳에 사본이 있다.

Data locality — 문서를 옮기지 말고 직원을 보내라

100GB 서류를 본사로 실어 오는 대신, 그 서류가 있는 창고로 작은 도장 (코드)을 든 직원을 파견한다. 옮기는 것은 데이터가 아니라 계산이다.

Sort-merge join — 정렬된 두 명단 나란히 맞추기

"출석부"와 "성적표"를 각각 학번 순으로 정렬해 나란히 놓고 위에서부터 내려오면 같은 학번끼리 자연히 만나 짝지어진다. reducer가 하는 일이 바로 이 "같은 key를 한자리에 모으기"다.

Hot key skew — 한 창구에만 늘어선 줄

유명인 계정 하나 때문에 특정 reducer 창구에 손님이 몰리면 그 줄만 끝없이 길어진다. 해결: 인기 손님을 여러 창구로 나눠 보내고(sharded join), 짝이 될 상대 서류는 각 창구에 복사해 둔다.

Broadcast hash join — 작은 전화번호부를 모두에게

큰 주문 목록을 나눠 맡은 직원들에게 작은 "고객 전화번호부"를 통째로 복사해 쥐여 준다. 각자 자기 주문을 보며 번호부에서 즉석 조회하면 되니 따로 모을(shuffle) 필요가 없다.

Materialization — 매 공정마다 창고에 다 쌓고 세는 공장

공정 A가 물건을 전부 만들어 창고에 쌓아야 공정 B가 시작한다. A의 마지막 한 개가 늦으면 B 전체가 대기한다. Dataflow engine은 컨베이어처럼 만들어지는 대로 다음 공정으로 흘려보내 창고 왕복을 없앤다.

RDD lineage — 요리 과정을 적어둔 노트

중간 결과를 냉장고에 백업(복제)하는 대신 "이 소스는 A를 볶고 B를 넣어 만들었다"는 조리 노트만 남긴다. 접시를 엎으면 노트를 보고 그 단계만 다시 만든다 — 단, 매번 같은 맛이 나오려면 레시피가 결정적이어야 한다 (deterministic).

Pregel/BSP — 라운드제 쪽지 교환

각 사람(정점)이 매 라운드(superstep)에 이웃에게 쪽지(메시지)를 보내고, 다음 라운드에 받은 쪽지로 자기 숫자를 고친다. 모두가 라운드 종을 기다렸다 동시에 다음 라운드로 넘어간다(동기적).

선언적 API — 목적지만 말하는 택시

"sort-merge로 조인해"라고 운전 방법을 지시하는 대신 "이 둘을 user ID로 붙여줘"라고 목적지만 말하면, 옵티마이저(기사)가 길(join 전략)을 알아서 고른다.

한 줄 요약

Batch processing의 사상은 불변 입력 → 통째로 새 출력이라는 Unix 파이프라인 철학(한 가지를 잘, 균일한 인터페이스로 조합)에서 출발해, 이를 HDFS + MapReduce로 수천 대에 확장하고(shuffle로 sort-merge join·group-by, 작은 쪽은 broadcast hash join, 계산은 데이터 가까이), 중간 상태를 매번 materialize해야 하는 MapReduce의 비효율은 dataflow engine(Spark/Tez/Flink — pipelined 실행 + lineage 재계산), Pregel/BSP 그래프 처리, 그리고 선언적 고수준 API(Hive/Pig/Spark SQL)가 극복한다.
